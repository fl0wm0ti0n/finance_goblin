# Purpose

BUG-0008 fixes subscription alert count mismatches and improves recurring-pattern detection from large transaction histories. After the fix, the **Subscriptions** page banner shows a **reconciled** unread count that matches pending patterns to review — not accumulated duplicate alerts from past syncs.

# Prerequisites

- US-0003 subscription engine running with synced Firefly transactions
- Successful sync to run detection (same as US-0003)
- BUG-0008 fix deployed (DEC-0071 + DEC-0072)

# What changed

## Alert count (defect W)

**Before:** Each sync could create another unread alert for the same pending pattern — the banner might show dozens of alerts while only a handful of patterns appear on the Pending tab.

**After:**

- Duplicate unread alerts are deduplicated automatically
- The banner uses a reconciled count: unread alerts linked to **pending** patterns only
- When you confirm or reject a pattern, related unread alerts are cleared
- The header notification bell (US-0005) is unchanged — subscription alerts stay on the Subscriptions page

## Detection recall (defect X)

**Before:** Long SEPA bank memo strings could split one merchant into multiple patterns; annual subscriptions might be missed with a one-year lookback.

**After:**

- Payee names are normalized (SEPA references stripped, legal suffixes collapsed)
- Bank transfers prefer the counterparty name when the memo is a transfer descriptor
- Detection looks back **730 days** (two years) for recurring patterns

More merchants you recognize should appear as pending patterns without flooding you with duplicate alerts.

# Usage steps

## 1. Sync and review

1. Run **Sync now** from Sync Status
2. Open **Subscriptions** → **Pending review**
3. Check the banner: unread alert count should match (or be less than) pending patterns shown
4. If the banner says "N pending patterns to review," that count comes from the pending tab — not stale alerts

## 2. Confirm or reject patterns

- **Confirm** or **Reject** as in US-0003 — unread alerts for that pattern clear automatically
- Rejected patterns stay excluded via fingerprint (unchanged US-0003 behavior)

## 3. Price-change alerts

- Price-change alerts are informational and separate from new-detection counts
- They deduplicate per pattern and amount direction — you won't get one per sync for the same change

# Troubleshooting

| Symptom | Likely cause | Action |
|---------|--------------|--------|
| Banner still higher than pending | Pre-fix duplicate alerts not yet backfilled | Re-sync after deploy; contact operator if persists |
| Fewer patterns than expected | Merchants need 2+ billing cycles in 730-day window | Wait for more history or check Firefly categories |
| Header bell shows 0 but banner shows alerts | Expected — subscription alerts are page-scoped | Use Subscriptions page banner |
