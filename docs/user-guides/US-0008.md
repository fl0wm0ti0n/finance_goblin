# Purpose

US-0008 adds **local and self-hosted AI providers** alongside OpenAI: Ollama (Compose full profile), LM Studio, LocalAI, and vLLM via an OpenAI-compatible HTTP API. The six-tool layer, PrivacyLayer, and audit redaction rules from US-0006 are unchanged.

# Prerequisites

- US-0006 operational (chat UI, audit log, privacy defaults)
- Migration `008_ai_audit_provider.sql` applied with prior migrations
- For Ollama: GPU or CPU host with enough VRAM for the chosen model
- For OpenAI cloud: `OPENAI_API_KEY` (or custom `api_key_env` name)

# Usage steps

1. Choose a provider mode in `backend/config/default.toml` (`openai`, `ollama`, or `openai_compatible`).
2. Set env vars from `.env.example` (`OPENAI_API_KEY` for cloud; optional `AI_*` overrides).
3. For Ollama: start `docker compose --profile full`, pull a model, set `provider = "ollama"`, restart backend.
4. For LM Studio / vLLM: set `openai_compatible` with required `base_url`, restart backend.
5. Open **Settings → AI & Privacy** — confirm provider table and run **Test AI provider**.
6. Open chat (Sheet or `/chat`) — confirm provider badge; ask a finance question; review tool transparency and audit **Provider** column.

# Example

Ollama on Compose full profile:

```toml
[ai]
provider = "ollama"
model = "qwen2.5:14b"
```

After `ollama pull qwen2.5:14b` and backend restart, **Test AI provider** returns latency and sample text; chat shows **Local · Ollama** without `OPENAI_API_KEY`.

# Limitations

- Restart required to switch provider, model, or `base_url` — no in-app hot-swap
- No in-app model catalog or pull UI — operator manages Ollama tags / vLLM server
- Local models may skip tools — warning SSE and optional nudge retry; no automatic OpenAI fallback
- Tool registry, PrivacyLayer, and six tools unchanged (AC4) — provider swap is HTTP layer only

# Provider modes

| `provider` | Label | API key | `base_url` |
|------------|-------|---------|------------|
| `openai` | Cloud · OpenAI | Required (`OPENAI_API_KEY` default) | Fixed `https://api.openai.com/v1` |
| `ollama` | Local · Ollama | Optional | Default `http://ollama:11434/v1` in Compose |
| `openai_compatible` | Local · Compatible | Optional | **Required** in TOML |

Env overrides (optional): `AI_PROVIDER`, `AI_BASE_URL`, `AI_MODEL`.

# Ollama setup (Compose full profile)

1. Start stack with Ollama:

```bash
docker compose --profile full up -d
docker compose --profile full exec ollama ollama pull qwen2.5:14b
```

2. Configure backend:

```toml
[ai]
provider = "ollama"
model = "qwen2.5:14b"
local_tool_nudge_retry = true
```

3. Restart the Flow Finance AI backend container/service.

4. Use **Settings → Test AI provider** and chat — badge should read **Local · Ollama**.

Use `--profile full` whenever `provider = "ollama"` so the `ollama` service is running. Compose cannot branch `depends_on` on env; document-only ordering hints apply.

# LM Studio / LocalAI / vLLM

**LM Studio** (host, backend in Docker):

```toml
provider = "openai_compatible"
base_url = "http://host.docker.internal:1234/v1"
model = "your-loaded-model"
```

**LocalAI** (sidecar example):

```toml
base_url = "http://localai:8080/v1"
```

**vLLM** — start server with tool support, e.g.:

```bash
vllm serve <model> --enable-auto-tool-choice --tool-call-parser llama3_json
```

Point `base_url` at `http://<host>:<port>/v1`.

# Settings — provider table and test

Open **Settings → AI & Privacy**:

| Column | Meaning |
|--------|---------|
| Provider | Human label (Local · Ollama, Cloud · OpenAI, …) |
| Model | Active model name from TOML |
| Base URL | Effective completions base (no secrets) |
| Status | Configured / Not configured |
| Test | Runs `POST /api/v1/ai/test` (minimal completion, no tools, no audit) |

**Test AI provider** results:

- Success: latency (ms) and short sample text
- Failure: actionable error (connection refused, model not found, HTTP body snippet)

Misconfigured OpenAI (missing key) or empty `base_url` for compatible mode shows **Not configured**; chat input is disabled.

# Chat — Local vs Cloud badge

The chat header (Sheet drawer and `/chat`) shows:

- **Privacy** badge (aggregates vs raw transactions)
- **Provider** badge from settings (`provider_label`)

When the model answers without calling tools on a local provider, you may see a yellow notice: local models sometimes skip tools; the backend may retry once with a nudge (`local_tool_nudge_retry`).

# Local tool reliability

- Prefer models with strong function/tool calling (`qwen2.5:14b`, `llama3.1:8b`).
- Local modes omit `tool_choice: auto` (Ollama/vLLM compatibility).
- Temperature defaults: **0.7** cloud, **0.3** local (override with `[ai] temperature`).
- There is **no fallback to OpenAI** when local fails (AC5).

# AC5 — local-only verification

**CI (no GPU):** `cargo test --test ai_local_provider_isolation` — wiremock proves chat hits only the configured local base URL.

**Operator UAT:**

1. Set `provider = "ollama"` (or compatible URL), unset `OPENAI_API_KEY`.
2. Restart backend, run Test AI provider and a chat question.
3. Confirm answers and tool rows work; packet capture or logs show no `api.openai.com` calls.

# Audit log — Provider column

Tool audit rows now include a **Provider** column (`openai`, `ollama`, `openai_compatible`). Older rows may have an empty provider (nullable migration).

# Troubleshooting

| Symptom | Likely cause |
|---------|----------------|
| Connection refused | Ollama/vLLM not running or wrong `base_url` / Docker network |
| Model not found | Tag not pulled (`ollama pull …`) or wrong `model` name |
| vLLM 400 on tools | Missing `--enable-auto-tool-choice` / parser flags |
| Not configured | Missing API key (openai) or empty `base_url` (compatible) |
| Weak factual answers | Model skipped tools — try a larger model or enable nudge retry |

# Boundaries

- Restart required to switch provider or model — no in-app pull or hot-swap
- Tool registry, privacy redaction, and six tools unchanged (AC4)
- Secrets stay in env only — never returned by settings or audit APIs
