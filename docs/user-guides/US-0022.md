# US-0022 — Deploy version stamp & stale-frontend detection

**Status:** Implemented (2026-06-14)  
**Sprint:** S0021  
**Acceptance:** AC-1, AC-2, AC-3, AC-4, AC-5, AC-6

## Overview

US-0022 eliminates operator deploy confusion by embedding build provenance into the UI and detecting stale frontend bundles after partial deploys. Operators can now confirm which release is running without `docker inspect` or behavioral guesswork.

## What changed

### 1. Backend metadata endpoint

**Endpoint:** `GET /api/v1/meta/build-info`

Returns compile-time build metadata:

```json
{
  "build_id": "a1b2c3d",
  "release_tag": "0.22.0-us0022",
  "build_timestamp": "2026-06-14T12:34:56Z"
}
```

- **Public route** (no auth required) — metadata is non-sensitive
- Uses `option_env!()` with fallback to `"dev"` / `"unknown"` — never breaks local dev
- Allowlist fields only — never echoes `.env` or PAT

### 2. Version stamp in sidebar footer

**Location:** `AppLayout` sidebar footer (below OIDC user name + logout)

**Appearance:**
- Short build id fragment (e.g., `a1b2c3d`) in monospace font
- Subtle gray color (`#94a3b8`), small font size (`0.7rem`)
- Always visible; low visual noise

**Hover tooltip:**
- Release tag (`__RELEASE_TAG__`)
- Build id (`__BUILD_ID__`)
- Build timestamp (UTC) — fetched from `/api/v1/meta/build-info` or embedded at compile time

### 3. Stale-frontend detection

**Mechanism:** On-mount fetch of `/api/v1/meta/build-info` with `cache: 'no-store'`

**Behavior:**
- Compares client `__BUILD_ID__` (embedded at compile time) to server `build_id`
- Mismatch → `stale=true` → non-blocking banner appears
- Match → no banner
- Skips when `__BUILD_ID__ === 'dev'` (local dev mode)
- Silent fail on network error (`.catch(() => {})`) — non-blocking

**Stale banner:**
- Yellow background (`#fef3c7`), dark text (`#92400e`)
- Message: "New version available — reload"
- Reload button triggers `window.location.reload()`
- Hidden when `!stale`
- Rendered above main content (below header)

### 4. Docker build-arg chain

**Dockerfile:** 3-stage build with `ARG` propagation

- **Builder stage (Rust):** `ARG` → `ENV` → `cargo build` (Rust `env!()` compile-time injection)
- **Frontend stage (Node):** `ARG` → `RUN BUILD_ID=$BUILD_ID npm run build` (Vite `define`)
- **Runtime stage (Debian):** `ARG` → `LABEL org.opencontainers.image.*` (OCI labels for `docker inspect`)

**CI invocation:**

```bash
docker build \
  --build-arg BUILD_ID=$(git rev-parse --short HEAD) \
  --build-arg RELEASE_TAG=$(cat handoffs/releases/latest-tag 2>/dev/null || echo "dev") \
  --build-arg BUILD_TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
  -t flow-finance-ai:latest \
  -f backend/Dockerfile .
```

### 5. Vite define injection

**File:** `frontend/vite.config.ts`

```typescript
define: {
  __BUILD_ID__: JSON.stringify(process.env.BUILD_ID || 'dev'),
  __RELEASE_TAG__: JSON.stringify(process.env.RELEASE_TAG || 'dev'),
}
```

- Static text replacements at compile time
- Works in both dev and production
- `.env` vars only work when `.env` file present

### 6. TypeScript declarations

**File:** `frontend/src/vite-env.d.ts`

```typescript
declare const __BUILD_ID__: string;
declare const __RELEASE_TAG__: string;
```

Ensures type safety for build-time constants.

## How to use

### Checking the running release

1. Open the app (any page)
2. Look at the sidebar footer (bottom of sidebar)
3. You'll see a short build id (e.g., `a1b2c3d`)
4. Hover over it to see:
   - Release tag (e.g., `0.22.0-us0022`)
   - Full build id
   - Build timestamp (UTC)

### What to do when I see the stale banner?

The stale banner appears when the frontend bundle is older than the backend (e.g., after a backend-only deploy).

**Action:** Click **Reload** to load the new version.

**Why it happens:**
- Backend was rebuilt with a new build id
- Frontend bundle still has the old build id
- On-mount detection compares client vs server build ids
- Mismatch → banner appears

**After reload:**
- New frontend bundle loads (with new build id)
- Client build id matches server build id
- Banner disappears

### Expected behavior after backend-only deploy

1. Operator deploys backend (new build id)
2. Frontend is not rebuilt (still has old build id)
3. Operator opens app (or reloads existing tab)
4. On-mount fetch detects mismatch → stale banner appears
5. Operator clicks **Reload** → new frontend bundle loads → banner disappears

This is the intended behavior — the banner tells the operator "the backend changed, reload to get the new frontend."

### Hard refresh hint (Traefik/browser cache)

If the stale banner persists after reload:

1. Hard refresh: `Ctrl+Shift+R` (Windows/Linux) or `Cmd+Shift+R` (Mac)
2. Or clear browser cache for the site
3. Or check Traefik cache TTL (if configured)

The tooltip shows the build timestamp — if it matches the deploy time, you're on the new version.

### Local development

In local dev (`npm run dev`):

- `__BUILD_ID__` and `__RELEASE_TAG__` default to `"dev"`
- Stale detection is skipped (no banner)
- Version stamp shows `dev` in sidebar footer
- Tooltip shows `Release: dev`, `Build: dev`, `Timestamp: <current time>`

This is intentional — local dev doesn't have compile-time build metadata.

## Troubleshooting

### Stale banner won't disappear after reload

**Symptom:** Banner persists after clicking **Reload**

**Possible causes:**
1. **Browser cache:** Hard refresh (`Ctrl+Shift+R` / `Cmd+Shift+R`)
2. **Traefik cache:** Check Traefik cache TTL (if configured)
3. **Frontend not rebuilt:** Verify frontend was rebuilt with new build id:
   ```bash
   docker inspect flow-finance-ai:latest | grep org.opencontainers.image.revision
   ```
4. **Backend not restarted:** Verify backend is running the new image:
   ```bash
   curl http://localhost:8080/api/v1/meta/build-info
   ```

### Version stamp shows `dev`

**Symptom:** Sidebar footer shows `dev` instead of a build id

**Cause:** App was built without `--build-arg BUILD_ID=...`

**Fix:** Rebuild with proper build args:

```bash
docker build \
  --build-arg BUILD_ID=$(git rev-parse --short HEAD) \
  --build-arg RELEASE_TAG=$(cat handoffs/releases/latest-tag 2>/dev/null || echo "dev") \
  --build-arg BUILD_TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
  -t flow-finance-ai:latest \
  -f backend/Dockerfile .
```

### Tooltip shows current time instead of build timestamp

**Symptom:** Tooltip timestamp is the current time, not the build time

**Cause:** Build timestamp was not embedded at compile time

**Fix:** Rebuild with `--build-arg BUILD_TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)`

### `/api/v1/meta/build-info` returns 401 Unauthorized

**Symptom:** Endpoint requires auth

**Cause:** Meta route is behind auth middleware

**Fix:** Meta route should be public (outside auth middleware). Verify `backend/src/lib.rs`:

```rust
let meta_router = meta::routes();

let mut router = Router::new()
    .merge(health_router)
    .merge(meta_router)  // <-- before api_router (which has auth)
    .merge(analytics_router)
    .merge(api_router)
```

## Acceptance criteria

- **AC-1:** Subtle stamp visible in sidebar footer; does not dominate primary UX
- **AC-2:** Hover tooltip reveals release tag + build id + build timestamp (UTC)
- **AC-3:** `GET /api/v1/meta/build-info` returns `{build_id, release_tag, build_timestamp}`; no secrets
- **AC-4:** Frontend bundle embeds build id at compile time (verify via `npm run build` + inspect bundle or `__BUILD_ID__` in DOM)
- **AC-5:** On-mount stale detection: mismatch → banner + reload CTA; match → no banner
- **AC-6:** `/health` liveness unchanged (`{status: ok}`); OIDC external profile smoke pass; no env secrets in metadata
