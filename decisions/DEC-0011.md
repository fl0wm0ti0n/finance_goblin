# Decision DEC-0011

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0002  
**Research:** R-0007

## Context

Precomputed forecast snapshots grow with each sync. Without retention, hypertable row count increases unbounded (accounts × ~730 daily points × sync frequency).

## Options

1. **Keep last 5 successful computations** — delete older metadata + cascade hypertable rows
2. **Time-based retention (30 days)** — harder to reason about sync frequency variance
3. **Keep only latest** — no audit trail for regression debugging
4. **No retention** — rejected for self-hosted long-running installs

## Recommendation

**Retain last 5 successful `forecast_computations` records.** After each successful recompute, delete older successful computations beyond the 5 most recent (by `computed_at`). Failed/running rows cleaned separately.

Configurable via `[forecast] retention_count` in TOML (default 5).

Hypertable rows reference `computation_id`; delete metadata rows triggers application-level cascade delete of forecast points for removed IDs.

## Impact

- Small cleanup step at end of `ForecastService::recompute`
- Bounded storage for typical self-hosted scale

## Risks

- Cascade delete on hypertables may be slow on first cleanup — acceptable at 5-computation cap
- Operators wanting longer history for US-0009 ML — increase config later
