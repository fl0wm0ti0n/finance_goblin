# Decision DEC-0010

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0002  
**Research:** R-0007

## Context

Acceptance requires forecasts recompute after Firefly sync completes. US-0001 sync uses a mutex (`active_run`) preventing overlapping syncs. Recompute may take seconds for multi-account × 24-month daily paths.

## Options

1. **Inline in sync task before clearing mutex** — extends lock duration; simplest coordination
2. **Separate async job queue (Redis)** — better isolation; premature without proven load (DEC-0005 deferred Redis dependency)
3. **Fire-and-forget spawn after sync** — race if second sync starts before recompute finishes
4. **Manual recompute only** — rejected (acceptance)

## Recommendation

**Extend sync mutex: recompute runs in the same Tokio task after successful entity sync, before `active_run` is cleared.**

Flow in `SyncService::execute_run`:

1. Firefly sync (existing)
2. On success: `ForecastService::recompute(sync_run_id)` 
3. On forecast failure: mark `forecast_computations` failed; sync run still `success` (ingest succeeded); API serves prior snapshot with `stale=true`
4. Clear `active_run` mutex

Manual `POST /api/v1/sync/trigger` returns 409 while sync **or** forecast recompute is active.

**Alternative for execute if latency exceeds ~30s:** spawn recompute but keep mutex until complete — same decision, implementation detail only.

## Impact

- Sync Status UI shows extended "running" state during recompute (acceptable)
- No Redis/job infrastructure in US-0002
- US-0003+ may introduce background queue if latency proven problematic

## Risks

- Long recompute blocks manual sync trigger — acceptable for self-hosted MVP
- Forecast failure after successful sync — operator sees stale forecast metadata; sync data still fresh
