# Decision DEC-0052

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0009  
**Research:** R-0050  
**Extends:** DEC-0010 (sync mutex), DEC-0023 (plan hook), DEC-0041 (exchanges phase)

## Context

ML forecast must stay paired with baseline from the same sync cycle for Compare AC. Plan refresh must bind to baseline per DEC-0023. ML failure must not regress US-0002 sync success semantics.

## Options

1. **Inline `forecast_ml` sub-phase after baseline, before exchanges** — recommended (R-0050)
2. **Async Tokio spawn after mutex release** — rejected (race with next sync; stale pairs)
3. **ML before plan hook** — rejected (plan must use baseline computation_id)
4. **Manual ML trigger only** — rejected (Compare AC needs fresh pair each sync)

## Recommendation

Pipeline order:

```text
subscriptions → forecast (baseline) → plan hook (baseline_id)
  → forecast_ml → exchanges → alerts
```

- Phase reporting: `"forecast_ml"` during ML sub-phase (Sync Status sub-label)
- ML error → record skip metadata on baseline row; **no failed ML computation row** (avoid retention pollution)
- Sync remains `success` when ingest + baseline succeed
- Alerts and wealth snapshot continue using **baseline** `computation_id`
- History gates (TOML): `min_monthly_points=12`, `min_portfolio_weeks=8`, sidecar healthcheck
- Combined baseline + ML + portfolio latency target **<30s**; sidecar timeout 60s

## Impact

- `backend/src/sync/mod.rs` — insert ML phase between forecast and `run_exchanges_and_alerts`
- `ForecastMlService::recompute(run_id, baseline_id)` — never propagates error to sync failure

## Risks

- Mutex duration growth when exchanges + ML combined — monitor sub-phase logs
- First-deploy numba JIT stall — acceptable once per deploy inside mutex
- Operator enables ML without `full` profile — `sidecar_unavailable` skip reason
