# Decision DEC-0015

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0003  
**Research:** R-0012, DEC-0004

## Context

Subscription state spans pending candidates, confirmed subscriptions, rejected patterns, price history, and alerts. Discovery asked: separate tables vs single polymorphic entity?

## Options

1. **Single lifecycle `subscription_patterns` + satellite tables** — per R-0012 schema
2. **Separate `subscription_candidates` + `subscriptions` tables** — duplicate columns; awkward promotion
3. **JSONB evidence blob in one table** — harder to query for Grafana
4. **Store confirm/reject in Firefly tags** — rejected (DEC-0004 read-only)

## Recommendation

**Single lifecycle table** `subscription_patterns` with `status` enum (`pending`, `confirmed`, `rejected`, `inactive`) and satellite tables:

- `subscription_pattern_transactions` — evidence link
- `subscription_price_events` — append-only billing and change history
- `subscription_rejections` — fingerprint PK for permanent exclusion
- `subscription_alerts` — page-scoped notifications

**Rejection semantics:** on reject, set pattern `status = rejected`, insert `subscription_rejections(fingerprint)`. Detection skips rejected fingerprints; forecast excludes them entirely (AC-8). Rejection is permanent for MVP (no un-reject).

**Fingerprint:** `hash(normalized_key + interval_days + round(median_amount, 2))` with UNIQUE constraint.

## Impact

- Migration `003_subscriptions.sql`
- Grafana queries join `subscription_patterns` + `subscription_price_events`
- CASCADE delete on pattern removes price history — acceptable for MVP

## Risks

- Fingerprint collisions if normalization too aggressive
- Rejection list grows unbounded — negligible at household scale
- No un-reject UX — operator can delete rejection row manually if needed
