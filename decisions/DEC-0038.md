# Decision DEC-0038

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0007  
**Research:** R-0033, R-0034  
**Extends:** US-0005 wealth analytics boundary

## Context

Acceptance requires realized gains, unrealized gains, and total return. Discovery asked FIFO vs exchange-reported PnL and tax reporting boundary.

## Options

1. **Hybrid: exchange fields for derivatives; avg-cost for spot; total return vs first-sync baseline** — per R-0033
2. **Full local FIFO tax lots** — rejected (tax-grade complexity; out of scope)
3. **Trust exchange PnL only** — rejected (Binance spot lacks single PnL field)
4. **Mark-only without realized** — rejected (acceptance requires all three metrics)

## Recommendation

**Scope:** wealth analytics only — not tax reporting.

| Metric | Primary | Fallback |
|--------|---------|----------|
| Unrealized | Exchange position PnL fields | Mark-to-market from avg-cost |
| Realized | Exchange cumulative + income ledgers | Local trade sum since watermark |
| Total return | `(current_eur − baseline_eur) / baseline_eur` | Baseline = first successful sync per exchange |

- Spot without exchange PnL: average-cost from imported trades
- Funding fees: separate ledger rows; included in realized subtotal with distinct UI line
- Reconciliation: prefer exchange cumulative when local sum diverges >1%; log warning on Sync Status
- Aggregate after EUR conversion (DEC-0039)

## Impact

- New `PortfolioEngine` module with `recompute_pnl` post-exchange-sync
- `portfolio_pnl_snapshots` and `portfolio_baselines` tables (migration 007)

## Risks

- First-sync 90-day backfill may miss older lots — document avg-cost limitation
- Cross-margin spot/futures transfers may distort per-wallet realized
