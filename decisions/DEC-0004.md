# Decision DEC-0004

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0001  
**Research:** R-0001

## Context

Acceptance criterion: "No write operations are issued to Firefly III API (read-only verified in integration test or audit log)." Firefly is the sole transaction source of truth; any accidental write could corrupt operator ledger data.

## Options

1. **GET-only typed HTTP client** — compile-time + runtime method guard; endpoint path allowlist
2. **Integration test with HTTP mock** — assert no non-GET outbound during sync
3. **Audit log table** — record every Firefly request (method, path, status)
4. **Documentation-only policy** — insufficient for acceptance verification

## Recommendation

**Layered enforcement (1 + 2 + 3):**

- `FireflyClient` wrapper accepts only GET; panics/returns error on other methods
- Path allowlist: `/api/v1/accounts`, `/transactions`, `/categories`, `/budgets`, `/tags`, `/piggy_banks` (+ pagination/search read paths as needed)
- `firefly_request_audit` table logs method, path, status_code, requested_at
- Integration test runs full sync against mock server; fails if any non-GET observed

## Impact

- Connector module is the only code path to Firefly API
- PAT stored server-side only (env/secret); never sent to browser
- Audit log enables operator review and acceptance evidence

## Risks

- Future Firefly API additions requiring POST for read operations (unlikely) would need explicit decision gate
- Audit log growth — rotate/truncate policy deferred to ops runbook
