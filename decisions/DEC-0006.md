# Decision DEC-0006

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0001  
**Research:** R-0003

## Context

React SPA and Axum API need authenticated communication. Options include BFF with server-side sessions vs stateless JWT validation.

## Options

1. **SPA + Bearer JWT** — frontend holds access token; backend validates via JWKS middleware
2. **BFF with PKCE session** — `axum-oidc-client` cookie sessions; backend proxies API
3. **API keys** — insufficient for multi-user OIDC requirement

## Recommendation

**SPA-first with stateless JWT validation (option 1).** `react-oidc-context` manages OIDC flow in browser. Axum middleware validates `Authorization: Bearer` against IdP JWKS. Public routes: `/health`, `/health/ready` only.

Manual sync `POST /api/v1/sync/trigger` is authenticated Flow API action — not a Firefly write.

## Impact

- No server-side session store required in US-0001
- CORS configured for SPA origin → API
- Token refresh handled client-side via silent renew

## Risks

- Access token in browser memory — standard SPA threat model; short-lived tokens + HTTPS required
- BFF may be reconsidered if US-0006 AI streaming needs cookie-based auth — decision gate at that story
