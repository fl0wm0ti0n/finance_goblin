# Decision DEC-0013

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0003  
**Research:** R-0009, R-0006, DEC-0007

## Context

US-0002 `forecast/recurring.rs` implements payee grouping, cadence detection, and amount tolerance for projection. US-0003 needs the same heuristics plus confidence scoring, persistence, confirm/reject, and Dauerauftrag classification. Discovery asked: reuse vs fork `detect_patterns`?

## Options

1. **Extract shared `recurrence` module** — forecast wrapper + subscription engine on common core
2. **Fork separate subscription engine** — duplicate algorithm logic
3. **Call forecast module directly from subscriptions** — couples subscription to forecast-specific types
4. **ML/DBSCAN clustering** — rejected (US-0009 scope; household volume too small)

## Recommendation

**Extract shared recurrence core** into `backend/src/recurrence/` (normalize, group, cadence, amount, detect). `forecast/recurring.rs` becomes a thin wrapper returning `RecurringPattern` for projection. `subscriptions/detection.rs` adds confidence tiers, persistence, and classification.

## Impact

- Refactor US-0002 `recurring.rs` during US-0003 execute (behavior-preserving)
- Single source of truth for cadence windows and tolerance logic
- Forecast and subscription detection stay aligned per DEC-0007

## Risks

- Refactor regression in forecast tests — run existing `recurring` unit tests after extraction
- Module boundary creep — keep subscription-specific logic out of `recurrence`
