# DEC-0107 — Scarcity evaluator JOIN column qualification (BUG-0018)

**Status:** Accepted  
**Date:** 2026-06-10  
**Work item:** BUG-0018 (acceptance **BE**, downstream **BF**)  
**Research:** [R-0088 §2](../docs/engineering/research.md#r-0088--bug-0018-evaluate_scarcity-ambiguous-balance--alert-eval-pipeline-abort)  
**Extends:** **DEC-0026** (alert evaluation rules per [R-0022](../docs/engineering/research.md#r-0022--alert-engine-evaluation-rules-scarcity-budget-drift-plan-viability)); **DEC-0028** (sync alerts phase per [R-0024](../docs/engineering/research.md#r-0024--post-sync-alert-engine-pipeline--net-worth-snapshot-hook))

## Context

Post-sync `AlertService::run_post_sync` invokes `evaluate_scarcity` **first**. The daily aggregate query JOINs `forecast_balance_daily fbd` with `accounts a` but uses unqualified `SUM(balance::float8)` and `ts::date` — both tables expose `balance` (`002_forecast_hypertables.sql`, `001_initial.sql`). PostgreSQL returns **42702** (`column reference "balance" is ambiguous`); the error propagates via `?` and aborts the entire wealth alert pass. Sync remains **success** (R-0024 warn-only). Downstream **BF**: empty `GET /api/v1/alerts`, header bell "No active alerts".

## Options

| Option | Verdict | Rationale |
|--------|---------|-----------|
| **A — Qualify `fbd.balance` and `fbd.ts`** | **Accepted** | One-line SQL fix; preserves projected-path household aggregate per R-0022; minimal diff |
| **B — Broader alias refactor across `evaluate.rs`** | **Rejected** | Sibling evaluators already unambiguous ([R-0088 §2](research.md#r-0088--bug-0018-evaluate_scarcity-ambiguous-balance--alert-eval-pipeline-abort)) |
| **C — Subquery filter instead of JOIN** | **Rejected** | Equivalent semantics; larger diff; harder to read |
| **D — Aggregate `accounts.balance`** | **Rejected** | Wrong semantics — Ist mirror balance, not projected forecast path |

## Decision

Accept **DEC-0107** as the scarcity SQL qualification contract:

| Contract element | Value |
|------------------|-------|
| File | `backend/src/alerts/evaluate.rs` — `evaluate_scarcity` only |
| SELECT | `fbd.ts::date AS day`, `SUM(fbd.balance::float8) AS balance` |
| WHERE date bounds | `fbd.ts::date >= $2 AND fbd.ts::date <= $3` |
| GROUP BY | `fbd.ts::date` |
| JOIN purpose | Filter included asset accounts only — aggregation reads **`fbd.balance` exclusively** |
| Forbidden | Unqualified `balance` or `ts`; `a.balance` or `SUM(a.balance::float8)` |
| Migration | **None** — runtime SQL string fix only |
| Sync semantics | **Unchanged** — preserve R-0024 warn-only; do not fail sync on alert eval error |
| Regression gate | `wealth_alerts_integration::wealth_snapshot_and_scarcity_alert_on_post_sync` PASS when `DATABASE_URL` set |

### Implementation sketch

```sql
SELECT fbd.ts::date AS day, SUM(fbd.balance::float8) AS balance
FROM forecast_balance_daily fbd
JOIN accounts a ON a.firefly_id = fbd.account_id
WHERE fbd.computation_id = $1
  AND a.type = 'asset'
  AND COALESCE((a.payload->>'include_net_worth')::boolean, true) = true
  AND fbd.ts::date >= $2 AND fbd.ts::date <= $3
GROUP BY fbd.ts::date
ORDER BY day
```

## Risks

| Risk | Mitigation |
|------|------------|
| Wrong column qualified (`a.balance`) | Code review + integration test asserts projected aggregation |
| CI skips integration test without DB | V1 operator gate + document `DATABASE_URL=… cargo test --test wealth_alerts_integration` |

| Sync fail-on-alert-error | **Reject** — violates R-0024 unless product overrides |
