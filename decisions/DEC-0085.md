# DEC-0085 — Payee+interval confirm inheritance (BUG-0015 Layer 2)

**Status:** Accepted  
**Date:** 2026-06-07  
**Work item:** BUG-0015 (acceptance **AU**, **AV**, **AW**)  
**Research:** [R-0081 §C](../docs/engineering/research.md#r-0081--bug-0015-confirmed-subscription-reconfirm-after-rebuild)  
**Extends:** **DEC-0015** (subscription schema), **DEC-0071** (alert dedup — necessary but insufficient alone), **DEC-0072** (normalization)

## Context

`upsert_pending_pattern` preserves `confirmed`/`rejected` only on **fingerprint** match. `load_confirmed_fingerprints` + `confirmed_fps.contains` skip exact hash only. When H1 drift produces a new fingerprint, detection INSERTs a fresh `pending` row and emits `new_detection` per DEC-0071 — bypassing confirm-once trust (US-0003).

## Decision

### 1. Confirm inheritance key (frozen)

Identity for operator confirm propagation:

```text
(payee_key, interval_days)   // after DEC-0084 normalization
```

**Not** payee-only — multiple subscriptions per merchant (e.g. two Apple tiers) require distinct `interval_days` or amounts at different cadences.

### 2. Repository maps (frozen)

| API | Contract |
|-----|----------|
| `load_confirmed_payee_intervals()` | All `status = confirmed` rows keyed by `(payee_key, interval_days)` → `{ id, fingerprint, … }` |
| `load_rejected_payee_intervals()` | All rejected patterns keyed by `(payee_key, interval_days)` |
| `merge_confirmed_pattern()` | UPDATE existing confirmed row — refresh amounts, dates, display; see DEC-0086 for fingerprint + tolerance |

### 3. Detection path (frozen)

In `run_candidates`, **after** exact fingerprint skip:

1. If `(payee_key, interval_days)` in rejected map (with DEC-0086 tolerance) → **skip**.
2. If matches confirmed map → **`merge_confirmed_pattern`** → **skip** pending upsert and **no** `new_detection` alert.
3. Else → existing `upsert_pending_pattern` pending path.

Retain `confirmed_fps` / `rejections` fingerprint sets for fast path and backward compatibility.

### 4. Stale inactive (AU4)

`mark_stale_inactive` must use **active payee+interval set** from current detection groups, not fingerprint set alone. Wire into `run_detection` post-candidates.

### 5. Index

Migration: `CREATE INDEX IF NOT EXISTS idx_subscription_patterns_payee_status ON subscription_patterns (payee_key, status)`.

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Skip without merge (option D) | Confirmed row drifts from live billing; orphan fingerprints — fallback only |
| Alert dedup by payee (option E) | Pending cards still show; violates AU |
| Reopen BUG-0008 | Distinct symptom; coordinate DEC-0071 only |

## Risks

- **Stale confirmed amount** if merge path bug — merge must update `current_amount` every detection run.
- **Duplicate rows pre-fix** — inheritance prevents new pending; orphan cleanup deferred.
- **Query cost** — index on `(payee_key, status)` mitigates; map loaded once per detection run.

## Implementation anchor

- `backend/src/subscriptions/repository.rs` — AU2
- `backend/src/subscriptions/detection.rs` — AU3, AU4
- `backend/src/subscriptions/service.rs` — load maps, wire stale pass
