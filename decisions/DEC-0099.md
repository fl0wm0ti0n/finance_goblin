# DEC-0099 — Manual confirm-from-discover (US-0020)

**Status:** Accepted  
**Date:** 2026-06-10  
**Story:** US-0020  
**Research:** [R-0085 §2](research.md#r-0085--us-0020-subscription-discover-majority-category--operator-tags)  
**Extends:** **DEC-0085** (payee+interval merge), **DEC-0086** (interval tolerance), US-0003 confirm persistence

## Context

US-0020 AC-2 requires operator confirm of searched candidates **without** auto-detection-only path. `POST /api/v1/subscriptions/:id/confirm` only transitions `pending → confirmed`; explorer candidates have no pattern row.

## Decision

### 1. Endpoint (frozen)

`POST /api/v1/subscriptions/discover/confirm`

```json
{
  "payee_key": "netflix",
  "interval_days": 30,
  "median_amount": -12.99,
  "transaction_ids": ["ff-1", "ff-2", "ff-3"],
  "kind": "subscription"
}
```

### 2. Server flow (single DB transaction)

1. Validate txs exist; share normalized `payee_key` (DEC-0084).
2. `fingerprint = compute_fingerprint(payee_key, interval_days, median_amount)`.
3. If `(payee_key, interval_days)` matches **confirmed** map → **`merge_confirmed_pattern`** (DEC-0085/0086); refresh row; link txs; recompute `display_category_id` (DEC-0100).
4. If matches **rejected** payee-interval → **409** — operator must clear rejection first.
5. Else **INSERT** `subscription_patterns` as **`confirmed`** directly (skip pending); link txs; set `confirmed_at`; compute `display_category_id`.
6. **Do not** emit `new_detection` alert (operator-initiated).

### 3. Response semantics

| Outcome | HTTP | Body hint |
|---------|------|-----------|
| Created | 201 | `{ pattern, merged: false }` |
| Merged | 200 | `{ pattern, merged: true }` |
| Rejected prior | 409 | payee-interval rejection message |
| Fingerprint conflict | 409 | same fail-safe as detection |

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Insert pending then existing confirm | Spurious pending card + alert noise; violates AC-2 |
| Extend `:id/confirm` with explorer payload | Overloads id-based route |
| 409 on payee+interval already confirmed | Conflicts with DEC-0085 merge (BUG-0015) |

## Risks

- Payload tampering with wrong `transaction_ids` — validate payee + interval tolerance on all txs
- Merge + majority drift — recompute display category on merge (DEC-0100)
