# Decision DEC-0027

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0005  
**Research:** R-0023  
**Extends:** R-0012 (subscription alert satellite pattern)

## Context

Unified Alert Engine inbox requires persistence with deduplication. Discovery asked: fingerprint vs cooldown; boundary with US-0003 `subscription_alerts`; Redis vs PostgreSQL dedup.

## Options

1. **PostgreSQL `alerts` table with fingerprint partial unique index** — per R-0023
2. **Merge subscription alerts into unified table** — rejected (backlog boundary; different UX surfaces)
3. **Redis SETNX dedup** — deferred (no app Redis dependency yet per R-0004)
4. **Permanent dismiss per entity+type** — rejected (user may miss re-emerging crisis)

## Recommendation

Migration 005 `alerts` table with enums: `alert_type` (`scarcity`, `budget_drift`, `plan_viability`), `alert_severity`, `alert_status` (`active`, `acknowledged`, `dismissed`, `resolved`).

**Dedup:** partial unique index on `fingerprint` WHERE `status IN ('active', 'acknowledged')`. On re-evaluate with same fingerprint → update `last_evaluated_at` + `context`; no duplicate insert.

**Lifecycle:**
- Condition clears → `status=resolved`, `resolved_at=now()`
- Acknowledge → sets `acknowledged_at`; removes from unread bell count
- Dismiss → `status=dismissed`; hide from bell + active list while condition persists; re-fire after clear + re-breach creates new row

**Boundary:** `subscription_alerts` unchanged. Optional header cross-link to subscription unread count only.

## Impact

- `backend/src/alerts/repository.rs` upsert/resolve/acknowledge/dismiss
- React `/alerts` and header bell query `alerts` table only

## Risks

- Monthly fingerprint may miss intra-month re-breach after resolve
- Acknowledged-but-active alerts need clear UI copy
