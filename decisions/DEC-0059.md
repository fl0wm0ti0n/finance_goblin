# Decision DEC-0059

**Status:** Accepted  
**Date:** 2026-06-05  
**Bug:** BUG-0006  
**Research:** R-0060  
**Extends:** DEC-0002 (sync upsert backfill), DEC-0005 (mirror `transactions.amount` semantics)

## Context

Firefly III transaction journal splits expose **positive** `amount` values; expense vs income direction is carried in split `type` (`withdrawal`, `deposit`, `transfer`). Flow Finance AI mirror and downstream SQL (transactions aggregates, subscriptions, forecast, alerts) assume **signed** amounts where **`amount < 0` means outflow**. Ingest today copies the raw positive split amount → category/month aggregates report `total_outflow: 0` while `transaction_count > 0`, and the AI `get_transactions` tool misleads the model (BUG-0006 Q3).

## Options

### 1. Normalize sign at ingest from split `type` (recommended)

In `sync_transactions`, before `upsert_transaction`, derive signed `amount` from the same first split leg used for scalar columns:

| Split `type` | Stored `amount` |
|--------------|-----------------|
| `withdrawal` | `-abs(amount)` |
| `deposit` | `+abs(amount)` |
| `transfer` | `-abs(amount)` on source leg (first split) |
| missing / unknown | `-abs(amount)` when raw `amount > 0`, else raw value |

Persist signed value in `transactions.amount`. Full journal unchanged in `payload` JSONB.

**Alternative:** Change every aggregate query to use `ABS(amount)` and read `type` from `payload` — rejected (cross-cutting duplication; subscriptions/forecast already use `amount < 0`).

**Alternative:** Negate only `withdrawal`, leave other types positive — rejected (incomplete fix for operator ledger).

**Alternative:** Account-role heuristic (source vs destination) — rejected for MVP complexity; first-split + `type` matches current ingest model.

### 2. Backfill existing mirror rows

Rely on DEC-0002 upsert + 7-day overlap on next Firefly sync — **no one-off SQL migration**.

**Alternative:** Operator SQL `UPDATE` script — rejected (extra runbook step; upsert sufficient).

## Recommendation

Accept option **1** with option **2** backfill path. Document in architecture § BUG-0006 and execute Q3 in sprint Q0010.

## Impact

- `TransactionsRepository` outflow/inflow CASE expressions work without change
- Subscriptions and forecast expense filters (`amount < 0`) align with mirror after re-sync
- Operator must run sync post-deploy to refresh historical rows

## Risks

- Transfer journals signed on first leg may mis-classify edge cases — mitigated by `period_status` (R1) and full `payload` retention
- Re-sync shifts subscription/forecast baselines — expected data correction, not regression
