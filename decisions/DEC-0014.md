# Decision DEC-0014

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0003  
**Research:** R-0009

## Context

Backlog requires confidence tiers 95/80/60% for subscription detection. Research mapped tiers to occurrence count, cadence stability, and amount tolerance bands. Need minimum emit threshold to control false-positive queue noise.

## Options

1. **Three tiers 95/80/60 with 60% minimum emit** — per R-0009 criteria table
2. **Binary high/low only** — simpler UX but loses Finanzguru-style graduated trust
3. **Emit all detected patterns** — rejected (60% tier exists to filter noise)
4. **Single confidence score 0–100 continuous** — rejected (acceptance specifies three tiers)

## Recommendation

**Confidence tiers:**

| Tier | Criteria |
|------|----------|
| 95 | ≥4 occurrences; stable cadence; amounts within ±5% |
| 80 | ≥3 occurrences; stable cadence; amounts within ±10% |
| 60 | ≥3 occurrences; stable cadence; ≤1 interval outlier OR amounts within ±15% |

**Do not emit** patterns scoring below 60%. Minimum 3 transactions spanning ≥2 billing cycles required for any tier.

Tolerances configurable via TOML `[subscriptions.confidence_tolerance_pct]`.

## Impact

- API and UI expose `confidence_pct` as 60, 80, or 95 only
- Pending queue contains only ≥60% candidates
- Alerts for `new_detection` fire only when pending row created

## Risks

- 60% tier still produces false positives — confirm/reject UX required before forecast override
- Annual subscriptions may never reach 95% with limited history — acceptable; user confirms
