# Decision DEC-0028

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0005  
**Research:** R-0024, R-0013  
**Extends:** DEC-0010, DEC-0018, DEC-0023

## Context

Alert evaluation must run after forecast and active-plan refresh complete. Discovery asked inline mutex vs async job. Combined sync pipeline latency already monitored under DEC-0018.

## Options

1. **Inline `"alerts"` phase after forecast (includes plan hook)** — per R-0024
2. **Async Tokio spawn after mutex** — rejected (race with next sync; stale evaluation)
3. **Alerts before forecast** — rejected (scarcity/plan viability need latest snapshots)
4. **Separate cron for alerts** — rejected (acceptance: post-sync proactive evaluation)

## Recommendation

Extend sync mutex pipeline:

```
sync → subscriptions → forecast (+ inline active plan refresh per DEC-0023)
     → alerts phase (net worth snapshot upsert + AlertService::evaluate)
     → clear mutex
```

- Phase lock value: `"alerts"` during snapshot + evaluation
- Sync Status UI: "Evaluating alerts…"
- Failure: log warning; sync run still `success` if ingest+forecast succeeded; preserve last alert state
- Defer Redis/async queue only if combined pipeline exceeds ~30s (DEC-0010 precedent)

## Impact

- `SyncService::execute_run` calls `AlertService::run_post_sync` before mutex release
- `AppState` gains `alerts: AlertService`, `wealth: WealthService`

## Risks

- Mutex duration grows ~100–500ms for alert pass — monitor combined latency
- Manual plan edits lag alerts until next sync — document in UI
