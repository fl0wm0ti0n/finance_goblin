# DEC-0074 — Plan-vs-actual 200 `no_active_plan` (BUG-0011 AF)

**Status:** Accepted  
**Date:** 2026-06-08  
**Work item:** BUG-0011 (sub-defect **AF**)  
**Research:** [R-0070](../docs/engineering/research.md#r-0070--bug-0011-planning-mode-compare-delta-empty-state-api-first-run-ux), [R-0017](../docs/engineering/research.md#r-0017--plan-vs-ist-daily-computation--aggregation-grain)  
**Extends:** DEC-0021 (plan-vs-Ist metric), US-0004 explicit Set active semantics  
**Mirror pattern:** `GET /api/v1/plans/risk-score` → `RiskScoreApiResponse::NoScore` (200 + tagged JSON)

## Context

`GET /api/v1/plans/active/plan-vs-actual` maps `PlanError::NoActivePlan` to **HTTP 404** `{ "error": "no_active_plan" }`. React `pvaQuery` has no error/empty branch — Plan vs Actual tab renders blank. Plans default `is_active=false` on create; operator must **Set active** (US-0004) — but raw 404 breaks the tab.

Research [R-0070](research.md#r-0070--bug-0011-planning-mode-compare-delta-empty-state-api-first-run-ux) §3 resolved empty-state contract. Acceptance **AF** requires **200 documented empty-state/guided UX** when no active plan.

## Decision

Return **HTTP 200** with tagged JSON when no active plan exists. Frontend renders guided empty state — mirror risk-score handling.

### 1. API response contract (frozen)

Tagged enum at route layer (`#[serde(tag = "status", rename_all = "snake_case")]`):

```json
// Active plan exists — existing payload shape under status=ok
{
  "status": "ok",
  "month": "2026-06",
  "reporting_currency": "EUR",
  "plan_stale": false,
  "actuals_stale": false,
  "rows": [ … ]
}

// No active plan — HTTP 200 (not 404)
{
  "status": "no_active_plan",
  "reason": "no_active_plan"
}
```

**Route:** `GET /api/v1/plans/active/plan-vs-actual` — handler checks active plan **before** calling `plan_vs_actual` service; on `None`, return `NoActivePlan` variant at **200**. Do **not** route `NoActivePlan` through `plan_error_status` (404).

**Rejected shapes:**

| Shape | Why rejected |
|-------|--------------|
| 404 + frontend-only fix | Acceptance **AF** requires API 200 |
| 200 + `rows: []` alone | Ambiguous vs active plan with sparse rows |
| Auto-activate on create | Violates US-0004 explicit Set active |

### 2. Frontend contract (frozen)

| Behavior | Contract |
|----------|----------|
| `pvaQuery` | `retry: false` on tab fetch |
| `status === "no_active_plan"` | Guided card: create plan + **Set active** CTA; link to Scenarios tab |
| `status === "ok"` | Existing chart + table unchanged |
| TypeScript | `PlanVsActualResponse` union type |

**Files:** `frontend/src/pages/PlanningPage.tsx` (and shared API types if extracted).

### 3. Breaking change notice

Clients expecting 404 on no active plan must migrate. Document in API changelog, user guide § Plan vs Actual, BUG-0011 release notes.

### 4. Auto-activate policy (frozen)

**Reject** global auto-activate on first plan create. Optional `plans.length === 1` shortcut **deferred** — prefer explicit Set active + AF guided state (R-0070 §4).

## Risks

| Risk | Mitigation |
|------|------------|
| API consumers rely on 404 | Changelog + release note |
| Ambiguous empty rows | Tagged `no_active_plan` distinct from `ok` |
| Operator skips Set active | Guided card + post-create banner (AD execute scope) |

## Acceptance traceability

- **AF** — `GET /api/v1/plans/active/plan-vs-actual` returns 200 JSON when active plan exists or documented empty-state when none; Plan vs Actual tab not broken by 404
- Footer — OIDC `/planning` tab smoke (all three tabs)

**Sequencing:** Ship with or after DEC-0073 AE backend; AD first-run banner complements AF guided UX.
