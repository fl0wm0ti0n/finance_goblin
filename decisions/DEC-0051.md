# Decision DEC-0051

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0009  
**Research:** R-0045, R-0043  
**Extends:** DEC-0007 (baseline boundary)

## Context

AC1 requires seasonal patterns detected and applied to monthly cashflow forecasts. Household net-cashflow series are short (often 12–36 monthly points). Model choice affects stability, explainability, and sidecar latency.

## Options

1. **Model selection ladder: AutoETS → MSTL → SeasonalNaive fallback** — recommended (R-0045)
2. **MSTL always** — rejected (<24 months overfit risk)
3. **Rule-based month-of-year factors only** — rejected (misses trend/level shifts)
4. **Apply seasonality inside DEC-0007 baseline engine** — rejected (violates baseline boundary)

## Recommendation

Sidecar model ladder based on monthly net-cashflow point count:

| History | Model |
|---------|-------|
| <12 monthly points | Skip ML (`insufficient_history`) |
| 12–23 points | AutoETS(`season_length=12`) |
| ≥24 points + `seasonal_strength ≥ 0.35` | MSTL(`season_length=[12]`, trend=AutoETS) |
| Fit failure | SeasonalNaive(`season_length=12`) + `low_confidence=true` |

Persist in computation metadata: `seasonal_periods`, `seasonal_strength`, `seasonal_detected`, `model_family`, `backtest_wmape` from sidecar cross_validation (h=6, n_windows=3 when ≥24 months available). Flag `low_confidence` when WMAPE > `wmape_low_confidence_threshold` (default 0.35).

Monthly tab UI reads metadata — no client-side seasonality engine.

## Impact

- Sidecar `/v1/forecast` implements ladder internally
- `ForecastMlService` passes point count + config thresholds
- React Monthly tab seasonal callout

## Risks

- False seasonal detection on sparse/noisy data — WMAPE gate + low_confidence UI
- Salary timing shifts break period-12 assumption — document in explainability panel
- MSTL ~2× latency vs AutoETS alone — acceptable within 30s mutex budget
