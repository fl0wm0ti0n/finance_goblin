# Decision DEC-0002

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0001  
**Research:** R-0002

## Context

Firefly III API has no native cursor or `updated_since` parameter. Incremental sync must be application-defined. Research found unreliable `updated_at` timestamps on transaction updates (GitHub #8282).

## Options

1. **Date-window overlap** — persist `last_successful_sync_at`; fetch with `start=<watermark - N days>`; upsert by Firefly `id`
2. **Search API `updated_at_after`** — narrower window but unreliable timestamp updates
3. **Full re-sync each run** — simple but unacceptable for large histories
4. **Firefly export endpoint** — CSV-only; not suitable for structured ingest

## Recommendation

**Date-window overlap with 7-day lookback** (configurable `sync.overlap_days`, default 7). Upsert all entities by Firefly resource `id`. Pagination: `limit=500`, page iteration until no `links.next`.

Reference entities (accounts, categories, budgets, tags, piggy banks): full-list sync each run.

Optional Search API refinement deferred to US-0002+ and never used as sole incremental signal.

## Impact

- `sync_cursors` table stores watermark per entity type
- Duplicate fetches within overlap window are expected; dedupe via upsert
- Initial sync may take minutes for large histories; Sync Status UI shows progress

## Risks

- 7-day overlap may miss edge cases if edits occur outside window without `updated_at` bump — mitigated by occasional full re-sync toggle (future operator config)
- Pagination bug history (#7427) — implement retry/backoff and page-count validation
