# DEC-0086 — Interval tolerance and fingerprint rotation on merge (BUG-0015)

**Status:** Accepted  
**Date:** 2026-06-07  
**Work item:** BUG-0015 (acceptance **AV**, **AW**)  
**Research:** [R-0081 §C/D](../docs/engineering/research.md#r-0081--bug-0015-confirmed-subscription-reconfirm-after-rebuild)  
**Extends:** **DEC-0085** (confirm inheritance), **DEC-0014** (confidence tiers — interval from recurrence engine)

## Context

Monthly subscriptions rarely land on exactly 30-day intervals (28–31 day calendar months). Strict `interval_days` equality would miss confirmed inheritance when detection recomputes interval slightly. Fingerprint rotation policy needed when merge updates a confirmed row — `fingerprint` column is UNIQUE and drives forecast export.

## Decision

### 1. Interval tolerance (frozen)

```rust
fn interval_matches(stored: i32, detected: i32) -> bool {
    (stored - detected).abs() <= 3
}
```

Apply when matching confirmed/rejected maps and stale-active sets. **±3 days** covers monthly cadence variance; weekly (~7d) and annual (~365d) patterns use same absolute window (acceptable — distinct merchants rarely collide within 3d at different cadences).

**Multi-subscription policy:** Two confirmed rows for same `payee_key` **must** differ by more than ±3 days in `interval_days` to coexist; otherwise operator must reject one — document in UAT.

### 2. Fingerprint rotation on confirmed merge (frozen)

When `merge_confirmed_pattern` runs:

1. Compute `new_fp = compute_fingerprint(payee_key, interval_days, median_amount)`.
2. **UPDATE** existing confirmed row **by primary key `id`**:
   - Set `fingerprint = new_fp`
   - Update `interval_days` to detected value
   - Refresh amount, confidence, display_name, last_seen_at
3. **Preserve** `status = confirmed`, `confirmed_at`, `id` (stable `pattern_id` for DEC-0071 alerts and UI).

**Rejected:** Keep legacy fingerprint on confirmed row — would re-break `confirmed_fps` skip on next run when amount drifts again.

**Rejected:** INSERT new row + delete old — unnecessary `pattern_id` churn breaks alert history.

### 3. Transaction boundary

Merge + fingerprint update + transaction relink in single DB transaction; on UNIQUE conflict (extremely rare hash collision), log and skip merge (fail-safe pending path).

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Percentage-based tolerance | Over-matches weekly vs monthly for borderline intervals |
| Payee-only match | Collapses multiple subs per merchant |
| No rotation — keep old fingerprint | Re-triggers H1 on next detection cycle |

## Risks

- **3-day window collision** — two real subs same merchant ±3d apart — operator must reject one; acceptable for MVP.
- **UNIQUE violation** on fingerprint update — transaction + fail-safe to pending path.
- **Forecast fingerprint export** — `load_confirmed_for_forecast` must read post-merge fingerprint (same row id).
