# DEC-0081 — Wealth all-holdings display and unified FX incomplete (BUG-0014 AQ)

**Status:** Accepted  
**Date:** 2026-06-09  
**Work item:** BUG-0014 (sub-defect **AQ**)  
**Research:** [R-0079 §6](../docs/engineering/research.md#r-0079--bug-0014-post-rebuild-omniflow-ml-sidecar-crypto-display-grafana-planning)  
**Extends:** **DEC-0064** (linear notional excluded from subtotal), **DEC-0080** (wallet + linear unrealized EUR), **DEC-0039** (FX), **R-0034** (FX incomplete banner)

## Context

Post-Q0020 omniflow probe: `holdings_count: 7`, `holdings_top: []`, `crypto.subtotal_eur: 0`, `unrealized_eur: 411.74`. Wealth breakdown hardcodes `unpriced_assets: []` and `fx_incomplete: false` regardless of PnL recompute output. Crypto tab holdings table renders `holdings_top` only (priced rows). Portfolio forecast `fx_incomplete_warning` uses a separate path — operator sees count **7** with empty table and divergent FX banners.

Research recommended a display contract; tier-2 `ExchangePriceBook` population remains deferred.

## Decision

### 1. API — `holdings_all` (frozen)

Add `holdings_all` to wealth breakdown response (cap **50** rows, stable sort: priced first by `value_eur` desc, then unpriced by `asset`):

| Field | Type | Contract |
|-------|------|----------|
| `asset` | string | Symbol or margin coin |
| `quantity` | number | Native contract qty or wallet balance |
| `product_type` | string | `spot` \| `futures` \| `linear` |
| `value_eur` | `Option<f64>` | `market_value_eur` when priced; `None` when unpriced |
| `unrealized_pnl_eur` | `Option<f64>` | From holding row; linear may have unrealized without `value_eur` |
| `native_unit` | string | Margin coin for futures wallet; contract symbol for linear |

**Retain `holdings_top`:** top-5 priced subset for backward compatibility — `holdings_all` is superset for UI table.

**Linear rows:** show native qty + symbol + unrealized EUR; **do not** add notional to `crypto.subtotal_eur` (DEC-0064).

### 2. Unified `fx_incomplete` (frozen)

```text
fx_incomplete = pnl.fx_incomplete || !unpriced_assets.is_empty()
unpriced_assets = pnl.unpriced_assets  // from compute_hybrid_pnl breakdown
```

Wealth API, WealthPage crypto tab, and portfolio-forecast warning **must** use the same flag and asset list.

**Linear with exchange unrealized:** excluded from `unpriced_assets` per DEC-0080 — banner must not fire when only linear unrealized is present and wallet is priced.

### 3. Explicitly deferred

- `ExchangePriceBook` population during recompute (tier-2 spot alts)
- Enum expansion for adjustment `target_type` (AS scope — DEC-0083)
- Merging linear unrealized into wealth subtotal (violates DEC-0064)

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Expand `holdings_top` in place without new field | Breaks consumers expecting priced-only top-5 |
| Frontend-only table from exchange API | Duplicates backend aggregation; no single FX gate |
| Price all linear notional into subtotal | Double-count with futures wallet per DEC-0064 |
| Tier-2 price book in this sprint | Scope creep; DEC-0080 deferred tier 2 |

## Risks

- Showing unpriced rows with `—` EUR may still confuse without help copy — AS2-style footnote on crypto tab acceptable in AQ2
- Cap at 50 may truncate large portfolios — acceptable for omniflow MVP
- `holdings_all` payload size — monitor; no pagination this sprint

## Implementation anchor

- `backend/src/wealth/service.rs` — wire PnL `unpriced_assets`; build `holdings_all`
- `backend/src/wealth/types.rs` — response types
- `frontend/src/pages/WealthPage.tsx` — table columns, unified banner
- `frontend/src/lib/api.ts` — TypeScript types
