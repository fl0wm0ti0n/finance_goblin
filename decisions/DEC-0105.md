# DEC-0105 — ai_tool_audit CHECK extension for forecast bucket audit (BUG-0017)

**Status:** Accepted  
**Date:** 2026-06-09  
**Work item:** BUG-0017 (acceptance **AY**, **AZ**)  
**Research:** [R-0087 §2](../docs/engineering/research.md#r-0087--bug-0017-post-sync-forecast-recompute-cluster-audit-check-fk-retention-ml-gate-forecastpage-loading)  
**Extends:** **DEC-0034** (migration 006 `ai_tool_audit` TEXT+CHECK pattern per [R-0030](../docs/engineering/research.md#r-0030--ai-tool-audit-log-persistence-migration-006)); **DEC-0078** (US-0015 bucket audit insert path)

## Context

US-0015 S0016 T-0173 ships `persist_bucket_audit` inserting `tool_name = 'forecast_bucket_assignment'` and `result_status` values `low_confidence`, `provider_unavailable`, `parse_error`. Migration `006_ai_audit.sql` CHECK constraints allow only six chat tools and `ok`/`error` statuses. Post-sync recompute logs WARN and drops audit rows — violating acceptance **AY** and **AZ** and reducing operator visibility per [R-0074](../docs/engineering/research.md#r-0074--us-0015-ai-forecast-bucket-mapping-rulellm-cascade-privacy) AC-6.

## Options

| Option | Verdict | Rationale |
|--------|---------|-----------|
| **A — New migration DROP+ADD both CHECK constraints** | **Accepted** | Matches project TEXT+CHECK convention; `NOT VALID` + `VALIDATE` safe on small audit table; preserves status fidelity |
| **B — Map extended statuses to `ok`/`error` in Rust** | **Rejected** | Loses operator distinction; fails **AZ** |
| **C — Native PostgreSQL ENUM** | **Rejected** | Type rebuild cost; CHECK is established pattern |
| **D — Separate `forecast_bucket_audit` table** | **Rejected** | Over-engineering; duplicates retention/query surface |

## Decision

Accept **DEC-0105** as the audit CHECK extension contract:

| Contract element | Value |
|------------------|-------|
| Migration file | New `0XX_bug0017_ai_audit_forecast.sql` (next sequential after existing migrations) |
| `tool_name` CHECK | Existing six chat tools **plus** `forecast_bucket_assignment` |
| `result_status` CHECK | `ok`, `error`, `low_confidence`, `provider_unavailable`, `parse_error` |
| Migration pattern | `DROP CONSTRAINT` → `ADD CONSTRAINT … NOT VALID` → `VALIDATE CONSTRAINT` per [R-0087 §2](research.md#r-0087--bug-0017-post-sync-forecast-recompute-cluster-audit-check-fk-retention-ml-gate-forecastpage-loading) |
| Rust insert path | **No change** — `bucket_inference.rs` / `service.rs` values already correct |
| Deploy order | Migration before backend image that inserts new values (standard SQLx ordering) |
| Pre-ship verify | Confirm constraint names on operator DB (`\d ai_tool_audit`) — names may differ from migration literals |

### Implementation sketch

```sql
ALTER TABLE ai_tool_audit DROP CONSTRAINT ai_tool_audit_tool_name_check;
ALTER TABLE ai_tool_audit ADD CONSTRAINT ai_tool_audit_tool_name_check
  CHECK (tool_name IN (
    'get_transactions','get_subscriptions','get_forecast',
    'get_budget_status','get_portfolio','simulate_plan',
    'forecast_bucket_assignment'
  )) NOT VALID;
ALTER TABLE ai_tool_audit VALIDATE CONSTRAINT ai_tool_audit_tool_name_check;

ALTER TABLE ai_tool_audit DROP CONSTRAINT ai_tool_audit_result_status_check;
ALTER TABLE ai_tool_audit ADD CONSTRAINT ai_tool_audit_result_status_check
  CHECK (result_status IN (
    'ok','error','low_confidence','provider_unavailable','parse_error'
  )) NOT VALID;
ALTER TABLE ai_tool_audit VALIDATE CONSTRAINT ai_tool_audit_result_status_check;
```

## Risks

| Risk | Mitigation |
|------|------------|
| Constraint name drift on operator DB | Pre-deploy `\d ai_tool_audit`; adjust DROP names if needed |
| Migration after code deploy | SQLx migration runs at startup before serving — standard ordering |
| Existing invalid rows block VALIDATE | Audit table small; inspect before VALIDATE; unlikely given inserts were rejected |
