# Decision DEC-0034

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0006  
**Research:** R-0030  
**Extends:** R-0004 (SQLx migration pattern)

## Context

Acceptance requires operator-visible tool call audit log. Discovery asked DB table vs log file and retention policy.

## Options

1. **PostgreSQL `ai_tool_audit` migration 006 + dual retention** — per R-0030
2. **JSONL log file only** — rejected (no Settings UI query without ingestion)
3. **Reuse `firefly_request_audit`** — rejected (different domain per DEC-0004)
4. **Unbounded retention** — rejected (storage minimization)
5. **Store full prompts/responses** — rejected (privacy layer purpose)

## Recommendation

**Schema:** `ai_tool_audit` with `session_id`, `user_subject`, `tool_name` (enum check), `args_summary` JSONB (redacted), `result_status`, `duration_ms`, `model`, `created_at`.

**Write path:** orchestrator inserts after each tool execution; `args_summary` via `PrivacyLayer::summarize_args`.

**Read path:** `GET /api/v1/ai/audit?limit=100&offset=0` — Settings AI & Privacy audit table.

**Retention (startup job):** keep newest **500 rows** AND delete rows older than **90 days** (configurable `[ai] audit_retention_days`, `[ai] audit_max_rows`).

**Immutability:** append-only; DELETE only via retention job.

## Impact

- Migration 006 required before chat execute
- Operator can correlate multi-tool turns via `session_id`

## Risks

- JWT `sub` not human-readable — optional OIDC display name later
- Retention DELETE vs compliance expectations — document in user guide
