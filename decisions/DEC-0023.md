# Decision DEC-0023

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0004  
**Research:** R-0019  
**Extends:** DEC-0010, DEC-0018

## Context

US-0004 plan projections must refresh when forecasts update and when users edit plans. Discovery asked: plan save only vs new sync mutex phase vs recompute all plans on sync.

## Options

1. **Plan save (async Tokio) + post-forecast hook for active plan only** — per R-0019
2. **Sync mutex `"planning"` phase** — rejected unless profiling > ~30s combined (DEC-0010 deferral pattern)
3. **Recompute all plans on sync** — rejected (wasteful)
4. **On-demand compute without persistence** — rejected (Grafana + acceptance)
5. **Active plan overlay on `/forecast`** — **deferred** to `/planning` only (R-0019)

## Recommendation

**Triggers:**

| Event | Action |
|-------|--------|
| Plan CRUD, template apply, new version, `set_active` | Spawn async `recompute_version` outside mutex |
| `ForecastService::recompute` success | Recompute active latest version with new `forecast_computation_id` |
| Sync mutex | No new phase |

Failure: plan recompute failure does not fail sync; serve last successful plan snapshot with `plan_stale=true`.

Optional link from `/forecast` to Planning UI — no dual overlay on forecast charts in US-0004.

## Impact

- Hook at end of `forecast::service::recompute`
- Plan API handlers spawn recompute after mutations
- Version-level guard against race with in-flight edits

## Risks

- User edits plan during post-sync recompute — use `is_latest` guard
- Plan succeeds on old baseline if forecast fails
