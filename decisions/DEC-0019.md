# Decision DEC-0019

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0004  
**Research:** R-0015, R-0007, DEC-0007

## Context

US-0004 requires scenario modeling (leasing, savings mode, house purchase, custom deltas) on top of US-0002 forecasts. Discovery asked whether to fork the forecast engine or build an independent projection layer.

## Options

1. **Delta overlay on latest forecast computation** — store adjustments only; apply at projection time (R-0015)
2. **Independent Plan Engine re-running full projection** — rejected (duplicates forecast logic; drift risk)
3. **Mutate forecast hypertables in-place** — rejected (breaks Ist baseline; violates read-only separation)
4. **Store only monthly deltas, derive daily on read** — rejected (Grafana + acceptance need stable daily grain)

## Recommendation

**Delta overlay** on the latest successful `forecast_computations` snapshot:

- **Current (Ist)** template = zero adjustments (baseline only)
- Built-in templates ship **preset adjustment lines** (Leasing +€300/mo, etc.) — user-editable before save; overridable via TOML
- **Savings mode** auto-suggests confirmed subscriptions (US-0003); user confirms selection before persisting removals
- Subscription-targeted removals match `payee_key` in overlay layer
- Output persisted to `plan_daily_cashflow` hypertable (separate from forecast tables)

## Impact

- New `backend/src/plan/{overlay,project,templates}` modules
- `PlanService::recompute_version` reads forecast baseline, writes plan series
- Forecast engine unchanged except post-recompute hook (DEC-0023)

## Risks

- Baseline stale when forecast recompute fails — surface `plan_stale` metadata
- Delta effective-date edge cases at month boundaries
- Multiple deltas targeting same payee need deterministic overlay order
