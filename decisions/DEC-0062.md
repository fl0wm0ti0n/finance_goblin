# Decision DEC-0062

**Status:** Accepted  
**Date:** 2026-06-05  
**Bug:** BUG-0005  
**Research:** R-0058, R-0059  
**Extends:** DEC-0037 (exchange connector read-only GET scope)

## Context

Bitunix ships two incompatible private REST auth schemes: spot on `openapi.bitunix.com` uses query-string `timestamp` + `sign` (`bitunix_sign`); futures on `fapi.bitunix.com` requires header auth (`api-key`, 32-char `nonce`, ms `timestamp`, `sign`) with a different SHA256 canonical string. The shipped connector implements spot only; futures sync is blocked (BUG-0005 **N**).

## Options

### 1. Dual-path client in `BitunixConnector` (recommended)

- Keep existing `signed_get` for spot unchanged.
- Add `bitunix_futures_sign` + `futures_signed_get` using `futures_base_url` config.
- Shared `ExchangeHttpClient` for GET transport and backoff.

**Alternative:** Host-sniffing single client — rejected (fragile; Binance uses explicit fapi host split).

**Alternative:** CCXT — rejected (DEC-0037 / R-0032).

**Alternative:** Futures-only rewrite — rejected (spot path works on omniflow; breaks spot-only regression).

### 2. Config surface

Add `futures_base_url = "https://fapi.bitunix.com"` to `[exchanges.bitunix]`; expose in settings view alongside `spot_base_url`.

## Recommendation

Accept option **1** with option **2**. Execute in Q0012 task **N1** before M1/N2/N4.

## Impact

- `backend/src/exchanges/http.rs` — new sign helper + unit fixture
- `backend/src/exchanges/bitunix.rs` — futures GET wrapper
- `backend/src/config/mod.rs`, `backend/config/default.toml` — `futures_base_url`

## Risks

- Sign spec drift vs Bitunix docs — mitigated by canonical-string unit test (R-0058)
- Accidental POST — unchanged GET-only audit on shared HTTP client
