# Decision DEC-0003

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0001  
**Research:** R-0005

## Context

Acceptance requires PostgreSQL never embedded in Compose (`database.mode = "external"`). Compose `depends_on: service_healthy` cannot gate on external database availability. App must tolerate DB startup ordering in dev.

## Options

1. **Startup retry loop** — exponential backoff in Rust backend until pool connects or timeout
2. **Compose init container** — wait-for-it sidecar; adds complexity, still doesn't verify TimescaleDB extension
3. **Fail fast** — immediate exit on connection refused; poor dev UX
4. **Embedded PostgreSQL** — violates acceptance constraint

## Recommendation

**Exponential backoff connection retry at backend startup**: initial delay 1s, max interval 8s, total budget ~60s. Log each attempt. `/health/ready` returns 503 until pool connected and migrations applied.

Linux dev: document `extra_hosts: ["host.docker.internal:host-gateway"]` for host-reachable DB.

## Impact

- Backend Dockerfile/entrypoint does not require external wait script
- Operator must configure `pg_hba.conf` / network for Docker bridge access
- Clear error message after timeout with remediation hints

## Risks

- 60s may be insufficient on slow cold-start hosts — make total budget configurable via env
- Migrations run after connect; TimescaleDB extension missing fails with explicit SQL error (operator prerequisite per R-0004)
