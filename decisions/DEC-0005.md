# Decision DEC-0005

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0001, US-0002  
**Research:** R-0004

## Context

Platform uses PostgreSQL + TimescaleDB. US-0001 needs mirror tables for Firefly ingest; US-0002 needs hypertables for forecast time-series. Schema boundaries must be clear to avoid premature complexity.

## Options

1. **Relational mirrors only in US-0001** — enable TimescaleDB extension; defer hypertables
2. **Hypertables from day one** — premature without forecast data model
3. **Separate databases** — Flow mirror vs analytics; operational overhead
4. **JSON files instead of DB** — insufficient for downstream engines

## Recommendation

**US-0001: relational mirror tables + sync metadata only.** First migration runs `CREATE EXTENSION IF NOT EXISTS timescaledb`. Hypertables for forecast storage created in US-0002 migrations.

Store raw Firefly JSONB snapshot alongside indexed columns for forward compatibility.

## Impact

- Migrations stay small and focused for foundation sprint
- US-0002 adds `create_hypertable()` without refactoring mirror schema
- Operator must install TimescaleDB on external PostgreSQL before first deploy

## Risks

- Extension missing causes migration failure — document prerequisite clearly
- JSONB storage size for large transaction volumes — acceptable for self-hosted scale; index on Firefly `id` and `date`
