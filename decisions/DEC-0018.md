# Decision DEC-0018

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0003  
**Research:** R-0013  
**Extends:** DEC-0010

## Context

US-0003 requires subscription detection after Firefly sync and forecast integration for confirmed/rejected patterns (AC-8). DEC-0010 established inline forecast recompute in sync mutex. Discovery asked: inline vs async queue?

## Options

1. **Inline subscriptions phase before forecast in same mutex** — per R-0013
2. **Detection after forecast** — rejected (one-cycle lag before confirmed subs affect projection)
3. **Async Redis job queue** — deferred until combined latency > ~30s
4. **Fire-and-forget spawn** — rejected (race with next sync per DEC-0010)

## Recommendation

**Extend sync pipeline order:**

```text
1. Firefly sync (phase: "sync")
2. SubscriptionService::run_detection (phase: "subscriptions")
3. ForecastService::recompute with DetectionResult (phase: "forecast")
4. Clear active_run mutex
```

Detection failure → log warning, forecast uses prior confirmed state from DB. Forecast failure → prior snapshot with `stale` (unchanged DEC-0010 semantics). Sync run `success` if ingest succeeded.

Sync Status API exposes `phase` values: `"sync"` | `"subscriptions"` | `"forecast"`.

Default detection window: last 12 months; optional weekly full rescan via config.

## Impact

- `SyncService` injects `SubscriptionService`
- `ForecastService::recompute` accepts optional `DetectionResult`
- Manual sync trigger blocked during all three phases
- Monitor combined duration; introduce queue only if proven problematic

## Risks

- Mutex duration grows O(transactions) for detection pass
- Long-running sync blocks manual re-trigger — acceptable for self-hosted MVP
- Phase UI must communicate extended "running" state
