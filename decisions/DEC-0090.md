# DEC-0090 — Category query index deferral policy (US-0018)

**Status:** Accepted  
**Date:** 2026-06-08  
**Story:** US-0018  
**Research:** [R-0083 §7](research.md#r-0083--us-0018-category-filters-expense-series-api--trend-analytics)

## Context

Expense-series SQL filters by `category_id` and date range over up to 24 months. Mirror has `idx_transactions_date` only — no `category_id` index. Research estimated &lt;10 ms on ~900 rows; architecture must decide whether to ship an index in US-0018.

## Decision

### MVP policy (frozen)

- **Do not** add `idx_transactions_category_date` in US-0018 execute by default
- Accept sequential scan for typical mirror scale (≤ few thousand rows, single category, 24-month window)

### Conditional gate (execute optional task P1)

During execute or QA on operator mirror (or CI fixture with representative row count):

```sql
EXPLAIN ANALYZE
SELECT ... -- expense_series_by_month query
```

If execution time **>50 ms** sustained:

- Add migration: `CREATE INDEX idx_transactions_category_date ON transactions (category_id, date)`
- Document in runbook; no Timescale continuous aggregate

### Explicitly rejected for MVP

- Materialized monthly rollup table
- Timescale `time_bucket` continuous aggregate refresh pipeline
- Pre-emptive index without EXPLAIN evidence

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Ship index unconditionally | Migration overhead without proven need on ~1k rows |
| Materialized view | Refresh orchestration overkill for AC-2 |
| Block US-0018 on index spike | Research confidence high at MVP scale |

## Risks

- Large mirrors (10k+ txs) may exceed 50 ms → P1 gate catches before release
- Index adds write amplification on sync ingest — acceptable only when proven necessary

## Implementation anchor

- Optional: `backend/migrations/00N_category_date_index.sql` (conditional)
- QA / operator EXPLAIN probe in sprint UAT
