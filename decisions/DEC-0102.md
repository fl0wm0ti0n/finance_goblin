# DEC-0102 — Subscription tag assign and list filter (US-0020)

**Status:** Accepted  
**Date:** 2026-06-10  
**Story:** US-0020  
**Research:** [R-0085 §4–5](research.md#r-0085--us-0020-subscription-discover-majority-category--operator-tags)

## Context

US-0020 AC-4 requires multiple tags per subscription and filtering the subscription list by tag. Tag schema is **DEC-0101**; assign/filter API contracts were open in R-0085.

## Decision

### 1. Assign API (frozen)

`PUT /api/v1/subscriptions/:id/tags`

```json
{ "tag_ids": ["uuid-1", "uuid-2"] }
```

- **Replace set** semantics — idempotent full assignment
- Pattern must be `confirmed` or `pending` (standing orders included)
- Unknown tag id → **404**

### 2. List filter (frozen)

Extend `GET /api/v1/subscriptions` query:

| Param | Behavior |
|-------|----------|
| `status` | unchanged |
| `kind` | unchanged |
| `tag` | slug match via join `subscription_pattern_tags` ⋈ `operator_tags`; **AND** with status/kind |

`account_id` list filter — **optional P2** if sprint tight; Discover has own account filter.

### 3. Response enrichment

List/detail pattern DTO includes `tags: [{ id, name, slug }]` when assigned — avoids N+1 from UI.

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| PATCH per-tag add/remove | More round-trips; replace-set simpler for drawer UI |
| Client-only tag filter | Breaks deep links and Grafana parity path |
| POST per assignment | Non-idempotent; harder to sync drawer state |

## Risks

- Join cost on list — acceptable at household scale (&lt;500 patterns)

## Implementation anchor

- `backend/src/api/subscriptions.rs` (list query + PUT tags)
- `frontend/src/pages/SubscriptionsPage.tsx` (chips + filter)
