# DEC-0106 — paired_baseline_id ON DELETE CASCADE and retention prune order (BUG-0017)

**Status:** Accepted  
**Date:** 2026-06-09  
**Work item:** BUG-0017 (acceptance **BA**; **BC** downstream)  
**Research:** [R-0087 §3](../docs/engineering/research.md#r-0087--bug-0017-post-sync-forecast-recompute-cluster-audit-check-fk-retention-ml-gate-forecastpage-loading)  
**Extends:** **DEC-0011** (forecast retention count); **DEC-0050** (ML overlay `paired_baseline_id`); **DEC-0052** (sync `forecast_ml` phase after baseline)

## Context

`enforce_retention()` in `forecast/repository.rs` deletes stale `forecast_computations` rows per `model_kind`, processing **baseline before ml_enhanced**. Migration `009_forecast_ml.sql` defines `paired_baseline_id` FK with default `NO ACTION`. Deleting a stale baseline while an `ml_enhanced` row still references it violates the FK → recompute `Err` → sync serves stale baseline per R-0050 warn-and-serve semantics. Acceptance **BA** fails; **BC** (plan stale) is downstream because plan hook runs only on successful recompute.

## Options

| Option | Verdict | Rationale |
|--------|---------|-----------|
| **A — `ON DELETE CASCADE` on `paired_baseline_id`** | **Accepted** | ML row is derivative of baseline; CASCADE auto-prunes paired ML when baseline pruned; robust against future delete paths |
| **B — Application pre-delete children** | **Rejected for MVP** | No schema change but must cover every delete entry point; higher maintenance |
| **C — Reorder loop ml_enhanced-first only** | **Rejected alone** | Non-stale ML can still reference stale baseline beyond retention window |
| **D — `ON DELETE SET NULL`** | **Rejected** | Orphan ML rows break compare API and meta pairing |

## Decision

Accept **DEC-0106** as the FK retention contract — **combo** of schema CASCADE **plus** application prune order:

| Contract element | Value |
|------------------|-------|
| Migration file | Same sprint migration file or companion `0XX_bug0017_forecast_fk_cascade.sql` |
| FK policy | `paired_baseline_id` → `forecast_computations(id)` **`ON DELETE CASCADE`** |
| Migration pattern | `DROP CONSTRAINT` → `ADD … ON DELETE CASCADE NOT VALID` → `VALIDATE CONSTRAINT` |
| Retention loop order | **`ml_enhanced` before `baseline`** in `enforce_retention()` — defense in depth |
| Rust delete logic | No explicit child DELETE required when CASCADE present |
| Sync semantics | **Unchanged** — recompute `Err` still warn-and-serve per R-0050; separate product gate deferred |
| BC resolution | Downstream of **BA** — re-smoke Planning Compare in verify-work only; no plan-engine change |

### Implementation sketch

```sql
ALTER TABLE forecast_computations
  DROP CONSTRAINT forecast_computations_paired_baseline_id_fkey,
  ADD CONSTRAINT forecast_computations_paired_baseline_id_fkey
    FOREIGN KEY (paired_baseline_id) REFERENCES forecast_computations(id)
    ON DELETE CASCADE NOT VALID;
ALTER TABLE forecast_computations VALIDATE CONSTRAINT forecast_computations_paired_baseline_id_fkey;
```

```rust
// enforce_retention — kind order frozen per DEC-0106
for kind in ["ml_enhanced", "baseline"] { /* existing DELETE logic */ }
```
