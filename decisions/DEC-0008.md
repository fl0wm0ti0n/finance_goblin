# Decision DEC-0008

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0002  
**Research:** R-0007, DEC-0005

## Context

Acceptance requires forecast data in TimescaleDB hypertables with recompute after sync. Discovery asked whether to precompute snapshots or compute on demand with cache.

## Options

1. **Precomputed snapshots on sync** — full series persisted; API/Grafana read latest computation
2. **On-demand compute with hypertable cache** — rejected (acceptance + Grafana stability)
3. **Single unified hypertable with metric_type** — fewer tables; awkward mixed granularities
4. **Relational tables without hypertables** — rejected (acceptance)

## Recommendation

**Precompute and persist** after successful Firefly sync (R-0007):

- Relational `forecast_computations` metadata (UUID id, sync_run_id, status, computed_at)
- Hypertable `forecast_balance_daily` — projected calendar date, account_id, balance
- Hypertable `forecast_cashflow_monthly` — month start, income/fixed/variable/free_cashflow
- Chunk intervals: 7 days (daily), 30 days (monthly)
- Indexes on `(account_id, computation_id, ts DESC)`

Adapt schema to US-0001 mirror keys: `account_id TEXT REFERENCES accounts(firefly_id)`, `sync_run_id UUID REFERENCES sync_runs(id)`.

## Impact

- SQLx migration `002_forecast_hypertables.sql` in US-0002 sprint
- API and Grafana query latest successful `computation_id`
- Operator TimescaleDB extension prerequisite unchanged from US-0001

## Risks

- Recompute latency scales with accounts × horizon days — see DEC-0010 mutex extension
- Failed computation must not overwrite last good snapshot — transactional write pattern
- External TimescaleDB missing extension — migration fails clearly
