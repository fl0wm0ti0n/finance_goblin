# Decision DEC-0040

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0007  
**Research:** R-0035  
**Extends:** US-0006 env-only secret pattern (`AiConfig.api_key_env`)

## Context

Acceptance requires API keys stored in self-hosted secrets/config, not transmitted externally. Discovery asked Settings runtime credential edit vs TOML/env.

## Options

1. **TOML metadata + env-only secrets; Settings read-only; test-connection endpoint** — per R-0035
2. **Encrypted secrets in PostgreSQL** — rejected (key management complexity)
3. **Settings form POST storing keys** — rejected (browser exposure)
4. **Docker secrets file mount only** — supported as operator option; env vars sufficient

## Recommendation

Mirror `AiConfig` / Firefly PAT pattern:

```toml
[exchanges.binance]
enabled = true
api_key_env = "BINANCE_API_KEY"
api_secret_env = "BINANCE_API_SECRET"
```

- Runtime: `ExchangeConfig::credentials()` reads env vars named in TOML
- Missing secret → `connection_state=not_configured`
- Never log, return, audit, or send secrets to OpenAI
- Settings UI: read-only enabled/configured badges + operator doc link
- `POST /api/v1/exchanges/{id}/test` — server-side read-only balance call

Key rotation requires env update + container restart (document operator flow).

## Impact

- TOML `[exchanges.*]` sections + Compose env template
- Settings Crypto exchanges section (display only)
- Integration test: secrets not in API responses or sync metadata

## Risks

- Operator commits secrets to git — document `.gitignore` + `.env.example`
- JWT session can trigger test/sync but cannot edit secrets — acceptable
