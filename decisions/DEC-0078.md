# DEC-0078 — AI-assisted forecast bucket mapping cascade (US-0015)

**Status:** Accepted  
**Date:** 2026-06-06  
**Story:** US-0015  
**Research:** [R-0074](../docs/engineering/research.md#r-0074--us-0015-ai-forecast-bucket-mapping-rulellm-cascade-privacy), [R-0075](../docs/engineering/research.md#r-0075--us-0015-forecast-bucket-privacy-feature-allowlist)  
**Extends:** DEC-0007 (category bucket config map), DEC-0032 (privacy defaults), DEC-0010 (inline recompute), US-0008 (`build_provider()`)

## Context

Discovery (`discovery-20260606-us0015`) confirmed BUG-0012 config baseline shipped in `categories.rs` / `project.rs`; gaps are AI inference for ambiguous recurring rows, monthly API provenance, AI-mapped UI badge, and forecast-path audit. Research (R-0074, R-0075) resolved six open questions: threshold, invalidation, privacy allowlist, API shape, provider reuse, and merchant-alias deferral.

US-0015 is **additive on the forecast projection path** — no chat tool registry changes (DEC-0069 / AC-7).

## Decision

Freeze the AI bucket mapping contract as eight elements below.

### 1. Cascade precedence (AC-1, AC-2)

| Stage | Mechanism | When |
|-------|-----------|------|
| 1 | DEC-0007 `[forecast.category_buckets]` + mirror `category_id` via `resolve_bucket` | Always first |
| 2 | In-module rule heuristics (sign/pattern short-circuit) | Config map miss on recurring due |
| 3 | LLM batch via `BucketInferenceService` | Ambiguous rows after stage 2; provider configured |
| 4 | Variable fallback | confidence &lt; threshold, provider absent, or HTTP failure |

**AC-1 guard:** Config-mapped Income/Fixed/Variable (including explicit `"variable"` in TOML) is **never** overridden by AI.

**Rejected alternatives:**

| Alternative | Why rejected |
|-------------|--------------|
| Chat `category_search` tool routing | DEC-0069 isolation; projection vs conversational trigger |
| RAG / vector merchant index | Over-engineering for MVP (R-0074) |
| Persistent Postgres assignment cache | Stale after sync edits; operator distrust (R-0074 §7) |

### 2. Confidence threshold

- Default **`ai_bucket_min_confidence = 0.75`** in `[forecast]` TOML (operator-overridable)
- `confidence >= threshold` → apply AI bucket to projection
- `confidence < threshold` → Variable + `low_confidence` audit row

**Rejected:** 0.90 default (excess Variable fallback on German merchant strings); 0.60 (under-safe for monthly card mutation).

### 3. Privacy feature allowlist (AC-3)

New `PrivacyLayer::prepare_bucket_features()` per **R-0075**:

- Sends: `category_name`, hashed `merchant_token`, `amount_sign`, `magnitude_band`, optional `recurring_label`, `pattern_class`
- Never sends under `allow_raw_transactions=false` (default): raw description/payee, IBAN, exact amount
- Opt-in `allow_raw_transactions=true`: normalized description for ≤50 rows/batch; document elevated risk
- **Same allowlist for local and cloud providers**

### 4. Invalidation

- Recompute inline each forecast pass (DEC-0010 post-sync)
- Hash `[forecast.category_buckets]` at inference start for config bust
- Optional in-call memo keyed by `payee_fingerprint + category_id + sign` within single `project_account` only
- **No cross-run DB assignment cache** for MVP

### 5. API provenance (AC-4, AC-5)

Extend `MonthlyPointResponse`:

```json
{
  "bucket_sources": {
    "income": "config",
    "fixed_costs": "ai",
    "variable_costs": "default"
  },
  "ai_mapped": true
}
```

- Per-bucket label = dominant source by mass; precedence `config` &gt; `ai` &gt; `default`
- `ai_mapped: true` when **any** AI-assigned mass in month (badge authoritative over mixed labels)
- Reject per-daily-point provenance

### 6. Provider integration

- Reuse US-0008 **`build_provider()`** — same provider/mode as chat
- No `forecast_ai_*` env split
- Provider absent → rule-only cascade (stages 1–2) + Variable fallback
- Provider HTTP error → skip LLM batch + `provider_unavailable` audit
- Batch cap: **100** feature rows per call

### 7. Audit trail (AC-6)

Persist `forecast_bucket_assignment` rows in `ai_tool_audit` (US-0006 pattern):

- Log `feature_id`, bucket, confidence, source, `rationale_code`
- Never log raw merchant/description

### 8. Deferred items (documented extension points)

| Item | MVP | Post-MVP |
|------|-----|----------|
| `[forecast.merchant_aliases]` regex table | Out | Stage between config and LLM |
| Rolling residual aggregate AI split | Variable only | New DEC if operator feedback requires |
| Cross-run assignment cache | Out | Optional stage-2 performance |

## Rolling residual (MVP limitation)

`variable_residual` daily rate accumulation stays **Variable** in MVP. AI enrichment applies to **recurring pattern dues** with ambiguous/missing categories. Stage-2 aggregate split requires separate architecture if acceptance pressure emerges.

## Risks

1. **Threshold calibration** — monitor `low_confidence` audit rate in QA
2. **Rolling residual** — operators may expect AI on discretionary spend mass; user guide must explain scope
3. **Provider cost** — batch ambiguous rows only; cap 100/call
4. **Privacy regression** — S1 code review on forecast path
5. **Mixed bucket_sources** — `ai_mapped` is badge source of truth
