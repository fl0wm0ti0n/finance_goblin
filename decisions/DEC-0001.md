# Decision DEC-0001

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0001  
**Research:** R-0003

## Context

US-0001 requires OIDC-protected UI with login redirect and session. Discovery left open the IdP choice (Authentik, Keycloak, or external). Minimal Compose profile must start without IdP infrastructure for acceptance testing.

## Options

1. **Authentik** — optional Compose `oidc` profile; modern admin UI; ~2–4 containers when embedded
2. **Keycloak** — heavier JVM footprint; better for enterprise LDAP/SAML federation
3. **Authelia** — lightweight forward-auth; narrower OIDC provider support for custom SPA
4. **External IdP only** — operator supplies issuer URL; no Compose IdP services

## Recommendation

**Authentik as default dev/prod IdP via optional Compose `oidc` profile**, with external IdP URL supported via `OIDC_ISSUER_URL` env (no embedded IdP required for minimal profile acceptance).

React: `react-oidc-context`. Axum: JWT validation via JWKS.

## Impact

- Minimal profile passes acceptance without Authentik running (operator may use mock/dev IdP or external URL)
- Documentation must cover both embedded Authentik setup and external IdP configuration
- Callback URLs and CORS must be documented per environment

## Risks

- Authentik adds operational overhead when enabled (2–4 containers, separate IdP DB)
- Callback URL mismatch causes silent login failures
