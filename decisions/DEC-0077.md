# DEC-0077 — Planning mutation feedback and error surface contract (US-0014)

**Status:** Accepted  
**Date:** 2026-06-08  
**Story:** US-0014  
**Research:** [R-0073](../docs/engineering/research.md#r-0073--us-0014-planning-mutation-error-toast-patterns)  
**Extends:** DEC-0073 (compare metric frozen), DEC-0074 (PVA API frozen), DEC-0024 (active plan + Dashboard 3)

## Context

Q0019 (BUG-0011) shipped functional planning paths — empty-state create, inline add form, Compare footnote, PVA guided card, partial success toast on Custom Apply only. Discovery (`discovery-20260608-us0014`) confirms **AC-7 gap**: seven `useMutation` hooks in `PlanningPage.tsx` have **no `onError` handlers** — API failures are console-only. AC-2/AC-5/AC-6 need success feedback and query invalidation polish; AC-1/AC-3/AC-4 are verify-only.

US-0014 is **frontend-only** unless execute finds a backend regression. No plans API contract changes.

## Decision

Freeze the planning mutation UX contract as five elements below.

### 1. Page-local feedback helper (not global MutationCache)

| Element | Contract |
|---------|----------|
| **Location** | `frontend/src/pages/PlanningPage.tsx` — module-local or co-located `planningFeedback.ts` in same directory |
| **API** | `showPlanningFeedback({ kind: 'success' \| 'error', message: string })` |
| **UI** | Reuse existing top-of-page card pattern; **success** background `#ecfdf5`; **error** background `#fef2f2` with optional Dismiss button |
| **Auto-dismiss** | Success: 4s (match Q0019 Custom Apply); Error: persist until Dismiss (operator must read failure) |
| **Message extraction** | From `Error.message` thrown by `apiFetch` — truncate to 240 chars; fallback `"Request failed"` |

**Rejected alternatives:**

| Alternative | Why rejected |
|-------------|--------------|
| Global `MutationCache` + toast library | No app-wide toast lib today; SubscriptionsPage uses same inline card; over-engineering for one page (R-0073) |
| Per-mutation duplicate `onError` bodies | Duplicates message parsing; violates DRY within PlanningPage |
| Inline field errors only | AC-7 requires operator-visible surfaces; network/500 failures need page-level feedback |

**Risks:** Toast flood on rapid retry — mitigate with single active feedback slot (new message replaces prior).

### 2. Mutation coverage (AC-7 — mandatory `onError`)

Every planning mutation **must** wire `onError: (err) => showPlanningFeedback({ kind: 'error', message: formatPlanningError(err) })`:

| Mutation | Endpoint | Default error label |
|----------|----------|---------------------|
| `createPlanMutation` | `POST /api/v1/plans` | "Could not create plan" |
| `activateMutation` | `POST /api/v1/plans/{id}/activate` | "Could not set active plan" |
| `applyTemplateMutation` | `POST .../apply-template` | "Could not apply template" |
| `createVersionMutation` | `POST .../versions` | "Could not create version" |
| `addAdjustmentMutation` | `POST .../adjustments` | "Could not add adjustment" |
| `updateAdjustmentMutation` | `PATCH .../adjustments/{id}` | "Could not update adjustment" |
| `deleteAdjustmentMutation` | `DELETE .../adjustments/{id}` | "Could not delete adjustment" |

Helper prepends label when body empty: `"Could not add adjustment: {detail}"`.

### 3. Success confirmation scope (AC-2, AC-5)

| Mutation | Success toast | Notes |
|----------|-----------------|-------|
| `createPlanMutation` | `"Plan \"{name}\" created"` | Empty-state **Create empty plan** + **Create from {template}** |
| `applyTemplateMutation` | Template-specific or `"Template applied"` | Extend beyond Custom-only (Q0019 AD3) |
| `addAdjustmentMutation` | `"Adjustment added"` | AC-2 — no silent success |
| `activateMutation` | `"Plan set as active"` | Clears set-active banner |
| `createVersionMutation` | `"New version created"` | Optional short toast |
| `updateAdjustmentMutation` | `"Adjustment updated"` | Optional — may omit if noisy |
| `deleteAdjustmentMutation` | `"Adjustment removed"` | Optional — may omit if noisy |

**Rejected:** Toast on every PATCH/DELETE — optional for edit/delete to avoid noise; **required** for create/add/activate paths per acceptance.

### 4. Query invalidation contract (AC-2)

After successful adjustment CRUD **and** activate **and** createPlan:

| Query key | Invalidate when |
|-----------|-----------------|
| `plan-version` | add / update / delete adjustment |
| `plan-compare` | add / update / delete adjustment; applyTemplate (existing) |
| `plan-vs-actual` | add / update / delete adjustment; activate; createPlan when tab may show PVA |
| `plans` | createPlan, activate (existing) |
| `plan-detail` | applyTemplate, createVersion (existing) |

**Rule:** Invalidate `plan-vs-actual` **immediately** on mutation success — do not wait for recompute badge to clear. Compare tab already invalidates on adjustment success.

### 5. Set-active banner copy (AC-6)

Extend existing yellow banner (`showSetActiveBanner`) to mention:

- Plan vs Actual tab requires an active plan (existing)
- **Grafana Dashboard 3 (`budgets`)** uses the active plan for plan-vs-ist panels (DEC-0024)

No auto-activate — frozen per DEC-0074.

## Consequences

- Execute adds ~1 helper module + mutation handler wiring — no backend changes
- OIDC smoke (AC-8) unchanged — pass-with-prerequisites **BACKEND_FRONTEND_DEPLOY**
- User guide documents first-run flow, Set active, Compare footnote semantics
- Future pages needing shared feedback should extract helper to `frontend/src/lib/` in a separate story — out of US-0014 scope
