# Decision DEC-0041

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0007  
**Research:** R-0036, R-0024  
**Extends:** DEC-0028 (sync alerts phase), DEC-0010 (inline mutex)

## Context

Exchange sync must feed crypto totals into net-worth snapshot before Alert Engine runs. Discovery asked pipeline order, parallel async, and independent exchange interval.

## Options

1. **Inline `"exchanges"` phase before `"alerts"`; independent interval + exchange-only partial run** — per R-0036
2. **Parallel async exchange job after mutex** — rejected (snapshot race; headline briefly excludes crypto)
3. **Exchanges after alerts** — rejected (violates acceptance — crypto must be in net worth snapshot)
4. **Fully independent cron outside mutex** — rejected (double snapshot complexity)

## Recommendation

Extended pipeline:

```
sync → subscriptions → forecast (+ plan hook) → exchanges → alerts → clear mutex
```

**Scheduler:**
- `[exchanges] interval_seconds` independent of Firefly (default same value)
- Exchange-only tick → phases 4–5 only (`exchanges` → `alerts`)

**Manual triggers:**
- `POST /api/v1/sync/trigger` — full 1–5
- `POST /api/v1/sync/exchanges/trigger` — 4–5 only

**Failure:** per-exchange error continues others; alerts run with partial data; `success_with_warnings` when ingest+forecast OK.

Defer async queue if combined pipeline exceeds ~45s (raised threshold for exchange fan-out).

## Impact

- `SyncService::execute_run` gains `"exchanges"` phase calling `ExchangeService::run_post_sync` then `PortfolioEngine::recompute_pnl`
- Sync Status UI: per-exchange rows + "Sync exchanges now"
- `WealthService::upsert_daily_snapshot` receives crypto EUR from portfolio engine

## Risks

- Mutex duration growth — log phase 4 timing separately; recommend longer exchange interval
- Exchange-only run uses last forecast for scarcity — acceptable
