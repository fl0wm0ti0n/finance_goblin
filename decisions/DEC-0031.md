# Decision DEC-0031

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0006  
**Research:** R-0027  
**Extends:** DEC-0004 (no direct DB from AI path), DEC-0006 (JWT auth on chat routes)

## Context

US-0006 requires OpenAI function-calling orchestration with exactly six registered tools. Discovery asked trait vs static schema, multi-round loops, and whether tools may access PostgreSQL directly.

## Options

1. **`AiTool` trait + static registry + async-openai Chat Completions loop** — per R-0027
2. **openai-func-enums proc-macro registry** — deferred (six tools manageable without extra dep)
3. **Assistants API with hosted threads** — rejected (external thread storage)
4. **Direct SQL in tool handlers** — rejected (acceptance + DEC-0004)
5. **Static schema files without trait dispatch** — rejected (needs Rust execution + testability)

## Recommendation

**Module:** `backend/src/ai/` with `orchestrator`, `registry`, `tools`, `provider` (stub for US-0008).

**Registry:** static `Vec<Arc<dyn AiTool>>` at startup; OpenAI `tools` array built from `parameters_schema()`.

**ToolContext:** holds cloned services (`ForecastService`, `SubscriptionService`, `PlanService`, `WealthService`, `AlertService`, `TransactionsService`) + `PrivacyConfig`. **No `DbPool`.**

**Loop:** stream completion → on `ToolCalls` validate args → execute (parallel when multiple) → privacy redact → truncate → append tool messages → repeat until `Stop` or **max_tool_rounds=5**.

**Config:** TOML `[ai]` for model, timeouts, rounds; API key from env only.

## Impact

- New `async-openai` dependency
- All six tools testable via mock `ToolContext`
- US-0008 adds provider implementations behind same orchestrator

## Risks

- Prompt injection via tool args — mitigate with serde + JSON Schema validation
- Runaway multi-round loops — cap at 5 rounds
- Token cost — cap tool output size (DEC-0035)
