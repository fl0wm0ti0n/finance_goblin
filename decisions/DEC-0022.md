# Decision DEC-0022

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0004  
**Research:** R-0018, R-0007

## Context

US-0004 requires persisted plans, versions, adjustments, and precomputed daily series for API, React, and Grafana. Discovery needed migration 004 schema boundaries.

## Options

1. **Relational plans + versions + adjustments + plan_computations + plan_daily_cashflow hypertable** — per R-0018
2. **JSONB blob per version** — rejected (Grafana SQL needs relational grain)
3. **Store plan series in forecast hypertables** — rejected (baseline isolation)
4. **No hypertable for plan_daily** — acceptable at scale but rejected for Grafana `$__timeFilter` parity with R-0007

## Recommendation

SQLx migration **`004_plans.sql`** with:

- `plans` + partial unique index `plans_one_active` (`is_active=true`)
- `plan_versions` with `version_number` 1–3, `is_latest`, `frozen_at`, `baseline_computation_id`
- `plan_adjustments` with direction/frequency/target enums
- `plan_computations` linking `forecast_computation_id`
- `plan_daily_cashflow` TimescaleDB hypertable (`ts`, `planned_net`, optional `planned_balance`)

Retention: last **3** successful computations per version.

## Impact

- Migration 004 in sprint execute
- Repository layer in `plan::repository`
- Cascade delete on plan/version removal

## Risks

- `baseline_computation_id` on frozen versions becomes informational stale reference
- Partial unique on `is_active` requires transactional activate-before-activate pattern
