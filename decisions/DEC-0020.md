# Decision DEC-0020

**Status:** Accepted  
**Date:** 2026-05-31  
**Story:** US-0004  
**Research:** R-0016

## Context

US-0004 acceptance requires plan versions v1, v2, v3 with side-by-side compare. Discovery asked: immutable snapshots on every save vs editable drafts vs unlimited version history.

## Options

1. **Hybrid: latest editable; freeze on "create new version"; cap at 3** — per R-0016
2. **Fully immutable every save** — rejected (too heavy for iterative what-if tuning)
3. **Unlimited versions** — rejected (acceptance caps at v1/v2/v3; compare UX breadth)
4. **Editable in-place with audit log only** — rejected (compare needs stable version boundaries)

## Recommendation

- `plan_versions.version_number` constrained to **1–3** per named plan
- Only `is_latest=true` version accepts adjustment edits
- **Create new version** copies adjustments from latest, sets `frozen_at` on prior version, increments version number
- v4 attempt returns **HTTP 409** with guidance to delete oldest or create a new named plan
- Compare API returns table-first metrics (monthly delta sum, projected month-end balance) + optional grouped bar chart data

## Impact

- `POST /api/v1/plans/{id}/versions` enforces cap
- `PATCH` on frozen version returns 409
- React Compare tab renders v1/v2/v3 columns

## Risks

- Users may expect v4+ for long-running plans — document cap in UX
- Frozen version references subscription IDs that may later be rejected
