# Decision DEC-0050

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0009  
**Research:** R-0043, R-0049, R-0050  
**Extends:** DEC-0007 (baseline algorithm), DEC-0011 (retention), DEC-0023 (plan hook)

## Context

US-0002 shipped a rule-based Forecast Engine (DEC-0007) as the authoritative forecast path. US-0009 adds ML-enhanced projections but must not replace baseline trust paths used by alerts, plan overlays, and AI defaults.

## Options

1. **`model_kind` discriminator on `forecast_computations` + layered overlay** — recommended (R-0049)
2. **Separate `forecast_ml_*` hypertables** — rejected (duplicate indexes; Grafana joins harder)
3. **Replace baseline with ML when history sufficient** — rejected (breaks DEC-0007, US-0002 AC, alert queries)
4. **`model_kind` only in metadata JSON** — rejected (Grafana `$forecast_variant` needs indexed column)

## Recommendation

- Add `model_kind TEXT NOT NULL DEFAULT 'baseline'` and `paired_baseline_id UUID` on `forecast_computations`
- ML pass produces **`model_kind=ml_enhanced`** row per sync cycle, linked via `paired_baseline_id` to baseline row from same `sync_run_id`
- ML integrates monthly deltas onto **baseline daily balance path** — never replaces DEC-0007 mechanics
- Retention: last **5 successful per model_kind** (extends DEC-0011)
- **Baseline authority:** alerts (DEC-0026), plan hook (DEC-0023), AI `get_forecast` default, Grafana default variant = `baseline`

## Impact

- Migration `009_forecast_ml.sql`
- `forecast::repository` — `latest_successful_by_kind(model_kind)`
- All existing queries without `model_kind` filter continue via default `'baseline'`

## Risks

- Operators confuse ML with authoritative forecast — Compare UI + docs required
- Retention doubles storage (5 baseline + 5 ML) — acceptable at household scale
- Plan hook must never bind to ML computation_id — enforce in code review
