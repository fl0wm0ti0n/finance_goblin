# Decision DEC-0049

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0009  
**Research:** R-0044, R-0043  
**Extends:** Compose `full` profile (Ollama pattern, R-0038), DEC-0010 sync mutex

## Context

US-0009 requires self-hosted seasonal statistical forecasting with prediction intervals. Operators on `minimal`/`standard` profiles must retain baseline-only behavior without errors. Research compared Python StatsForecast sidecar, embedded Rust `augurs`, and subprocess CLI.

## Options

1. **Python StatsForecast FastAPI sidecar in Compose `full` profile** — recommended (R-0044)
2. **Embedded Rust `augurs`** — rejected (incomplete seasonal ETS, no cross_validation parity)
3. **Subprocess CLI wrapper** — rejected (no healthcheck, ops fragility)
4. **External cloud ML APIs** — rejected (backlog explicit out-of-scope)

## Recommendation

Add **`stats-forecast`** service:

- Compose **`profiles: [full]`** only; internal URL `http://stats-forecast:8090`
- Backend env `STATS_FORECAST_URL`; TOML `[forecast_ml] enabled = false` **by default**
- Contract: `POST /v1/forecast` with monthly/weekly series, `level=[90]`, `model=auto`
- Backend HTTP timeout **60s**; healthcheck `GET /health`
- When disabled or unreachable: skip ML pass with metadata reason — baseline unaffected

Mirror Ollama operator pattern: document `--profile full` + `enabled = true` in user guide.

## Impact

- New `stats-forecast/` directory (FastAPI + Dockerfile)
- `docker-compose.yml` — `stats-forecast` service on `full` profile
- `backend/src/forecast_ml/sidecar.rs` — HTTP client
- `config` — `[forecast_ml]` section

## Risks

- Second Python runtime in `full` profile (~180–250 MB image)
- numba JIT cold start on first sync after deploy
- Operator forgets `full` profile — document clearly; default `enabled=false` prevents errors on minimal
