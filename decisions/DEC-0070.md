# DEC-0070 — Root README living documentation contract (US-0016)

**Status:** Accepted  
**Date:** 2026-06-08  
**Story:** US-0016  
**Research:** [R-0067](../docs/engineering/research.md#r-0067--us-0016-root-readme-research-template-parity-product-status-maintenance-hooks), [R-0066](../docs/engineering/research.md#r-0066--root-readme-split-layout-and-living-doc-maintenance)  
**Extends:** doc-profile split layout (US-0077 / DEC-0059 doc-profile record in runbook § documentation profile validation)

## Context

The repository lacks a root `README.md`; `validate_doc_profile.py` fails with `README.md missing`. Operators and contributors need a DEC-0059 split-layout entry document that stays current as US/BUG items close — without per-commit automation or backlog duplication.

Discovery and research resolved three open questions ([R-0067](research.md#r-0067--us-0016-root-readme-research-template-parity-product-status-maintenance-hooks)):

1. **`template/` tree absent** — partial stub rejected; full mirror required for parity.
2. **Product status placement** — validator H2 budget counts required `USER_*` H2s only; scannability favors a capped subsection under Purpose.
3. **Maintenance binding** — phase-boundary hooks at release and refresh-context, not per-commit.

Active profile: `DOC_AUDIENCE_PROFILE=both`, `DOC_DETAIL_LEVEL=balanced` (merged scratchpad).

## Decision

Freeze the US-0016 documentation architecture as three contracts below. No application code changes in this decision.

### 1. Template parity posture

| State | Validator command | Rationale |
|-------|-------------------|-----------|
| `template/` **absent** (current) | `python scripts/validate_doc_profile.py --repo . --no-template-parity` | Skips `[DOC_TEMPLATE_PARITY_FAIL]` until full tree ships ([R-0067](research.md#r-0067--us-0016-root-readme-research-template-parity-product-status-maintenance-hooks) §1) |
| `template/` **present** | `python scripts/validate_doc_profile.py --repo .` (default) | Requires `template/README.md` **and** `template/docs/developer/README.md` with matching DEV H2 presence |

**Rejected alternatives:**

| Alternative | Why rejected |
|-------------|--------------|
| Stub `template/README.md` only | Parity also requires dev shard; half-stub adds drift |
| Default validator without template | Fails closed once root README exists |
| Defer `--no-template-parity` removal indefinitely | Silent parity gap when template lands |

**Flip gate:** Drop `--no-template-parity` in the **same change set** that adds the full `template/` mirror. Document flip in runbook § README maintenance (US-0016).

**Risks:** Flag left on after template ships → mitigated by runbook note + release checklist. Operator runs default validator locally → mitigated by documenting both commands.

### 2. Product status placement and content

| Contract | Value |
|----------|-------|
| **Placement** | `### Product status` under `## Purpose` in root `README.md` |
| **Format** | Reverse-chronological bullets: `{US-xxxx\|BUG-xxxx} — {one-line outcome}` |
| **Cap** | **8** entries (drop oldest) |
| **History link** | `docs/product/backlog.md` for full history |
| **Forbidden** | Dedicated `## Product status` H2; nesting under Related documentation; acceptance-table dumps; secrets |

**Rejected alternatives:**

| Alternative | Why rejected |
|-------------|--------------|
| `## Product status` H2 | Extra TOC noise; no validator benefit over H3 ([R-0067](research.md#r-0067--us-0016-root-readme-research-template-parity-product-status-maintenance-hooks) §2) |
| `###` under Related documentation | Semantically wrong (status ≠ link index) |
| Per-commit auto-update | Noisy; rejected by R-0066 |

### 3. Maintenance hooks (phase boundaries)

Living-doc updates bind to **release** and **refresh-context** only.

#### Release (`/release`)

After backlog reconciliation (≈ step 10), before runbook readiness (≈ step 14):

> When any **US** or **BUG** in the target sprint transitions to **DONE** / **CLOSED**, append one bullet to root `README.md` **`### Product status`** in the form `{id} — {one-line outcome}`; trim to the **8** most recent entries. Before finalizing release readiness, run `python scripts/validate_doc_profile.py --repo . --no-template-parity` (drop `--no-template-parity` only after `template/` tree exists); non-zero exit → fail closed with remediation pointing to runbook § README maintenance (US-0016).

#### Refresh-context (`/refresh-context`)

After backlog status reconciliation:

> When release or sprint artifacts closed a **US** or **BUG** since the prior refresh, verify root `README.md` **`### Product status`** includes the closed id(s); update if missing. When README or doc-profile surfaces were touched, run `python scripts/validate_doc_profile.py --repo . --no-template-parity`.

#### Developer shard pointer

`docs/developer/README.md` § Workflow or Quality gates — one sentence pointing contributors to runbook § README maintenance after `/release` or `/refresh-context`.

#### Runbook (execute)

New subsection **`README maintenance (US-0016)`** under § documentation profile validation — embed the three hooks above; document `--no-template-parity` posture and flip gate.

**Risks:** Stale Product status → mitigated by release fail-closed validator + refresh-context verify step.

## Split layout (unchanged — execute implements)

Root `README.md` under `(both, balanced)`:

| Element | Requirement |
|---------|-------------|
| User H2s | `Purpose`, `Quickstart`, `Examples`, `Limitations`, `Related documentation` (exact titles) |
| Contributing | Single `## Contributing` → `docs/developer/README.md` |
| Forbidden in root | Any `DEV_*` H2 titles |
| Optional crosslinks | `USER_GUIDE_MODE=1` → mention `docs/user-guides`; `SPEC_PACK_MODE=1` → mention engineering/spec paths |
| Dev shard | `DEV_*` sections remain in `docs/developer/README.md` only |

Content sources: product value, compose profiles from `.env.example`, runbook operator commands ([R-0066](research.md#r-0066--root-readme-split-layout-and-living-doc-maintenance)).
