# DEC-0067 — Component-level monthly forecast bucket attribution

**Status:** Accepted  
**Date:** 2026-06-05  
**Bug:** BUG-0012  
**Research:** [R-0063](../docs/engineering/research.md#r-0063--bug-0012-forecast-monthly-bucket-component-attribution)  
**Extends:** DEC-0007 (hybrid forecast; category→bucket map)

## Context

BUG-0012 AG/AH: monthly Income and Fixed buckets are permanently zero on `/forecast` Monthly tab despite categorized mirror transactions. Root cause: `project_account` assigns **one bucket per calendar day** from **net daily `delta` sign** via `categorize_delta`, ignoring `category_names` and recurring category semantics. DEC-0007 intent (R-0006 §4) requires monthly decomposition by **projected component**, not net delta.

## Decision

**Component-level monthly attribution** for `monthly_map` accumulation only:

| Daily component | Monthly bucket rule |
|-----------------|---------------------|
| `rolling.daily_rate` | **Variable** (`variable_costs` if negative; `variable_costs` if positive — uncategorized residual inflow) |
| Each recurring due amount | `resolve_bucket(pattern.category_id, category_names, config)` → `map_category` on **lowercased category name** |
| Internal transfers | Excluded (unchanged) |

**Daily balance path unchanged:** `balance += delta` where `delta = rolling.daily_rate + Σ recurring_due`; milestones, horizons, and `forecast_balance_daily` writes are untouched.

**Unmapped categories** (no TOML key for resolved name): default to **Variable** per existing `map_category` fallback.

**Income bucket** is populated only when a recurring component resolves to an income bucket via category map — not from positive net daily delta alone.

## RecurringPattern category carry (frozen)

1. Add `category_id: Option<String>` to `RecurringPattern`.
2. In `detect_patterns`, set from **mode** of non-null `RecurrenceGroup.category_ids` (tie-break: latest transaction date).
3. In `apply_subscription_override`, when pushing `ConfirmedRecurring`: inherit `category_id` from replaced heuristic pattern, or **runtime lookup** from mirror transactions matching normalized `payee_key` when override lacks category.
4. On due-day, bucket recurring amount via resolved category — never description-only or net-sign routing.

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Fix sign check only (`delta >= 0` → Income) | Still one bucket per day; Fixed stays 0 (AH) |
| Re-bucket net delta using dominant category of day's txs | Fragile; loses recurring semantics |
| Full transaction replay for monthly totals | Duplicates projection layers; drift + performance risk |
| Positive `rolling.daily_rate` → Income | Uncategorized misc inflow must not masquerade as salary; Income requires category map |

## Operator TOML alignment (out of code scope)

German/custom Firefly category labels (e.g. `Gehalt`, `Miete Nebenkosten`) require operator extension of `[forecast.category_buckets]` in TOML — document in runbook omniflow checklist. Fuzzy/AI mapping deferred to **US-0015**.

## Risks

| Risk | Mitigation |
|------|------------|
| Variable totals shrink when fixed recurring moves to Fixed | Intended DEC-0007 behavior; unit regression test |
| Omniflow labels miss default TOML keys | Runbook checklist; verify-work AG/AH after operator TOML extend |
| Plans/alerts consumers assumed all-Variable outflow | Bucket semantics become correct; no API shape change |
| ML overlay (US-0013) | Unchanged; baseline `model_kind` only |

## Implementation anchor

- `backend/src/forecast/types.rs` — `RecurringPattern.category_id`
- `backend/src/forecast/recurring.rs` — mode carry from `RecurrenceGroup.category_ids`

- `backend/src/forecast/service.rs` — pass `category_names` into projection (already loaded)
- `backend/src/recurrence/detect.rs` — source `category_ids` (existing)
