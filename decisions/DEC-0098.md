# DEC-0098 — Subscription discover explorer API (US-0020)

**Status:** Accepted  
**Date:** 2026-06-10  
**Story:** US-0020  
**Research:** [R-0085 §1](research.md#r-0085--us-0020-subscription-discover-majority-category--operator-tags)  
**Extends:** US-0003 (`detect_recurrence_groups`), **DEC-0084** (payee_key), **DEC-0086** (`interval_matches`)

## Context

US-0020 AC-1 requires operators to search subscription **candidates** by account, payee text, and repeating interval. No discover API or Discover tab exists; detection runs only on sync via `DetectionPipeline::run_candidates`.

## Decision

### 1. Explorer engine (frozen)

Reuse Rust recurrence core — **not** ad-hoc SQL `GROUP BY payee_key`:

| Step | Contract |
|------|----------|
| Load | `load_expense_transactions(window_days)` with SQL push-down for `account_id` + `date >= cutoff` (`SubscriptionsConfig.detection_window_days`, default 365) |
| Group | `detect_recurrence_groups(&txs, &RecurrenceConfig::default())` |
| Filter | Post-group: payee `ILIKE`; interval bucket via `interval_matches` (DEC-0086 ±3d); optional `amount_min`/`amount_max` on `median_amount` (**P2 stretch**) |
| Exclude | Drop fingerprint ∈ confirmed ∪ rejected; drop `(payee_key, interval_days)` in confirmed/rejected payee-interval maps (same as `run_detection`) |
| Cap | Sort `confidence_pct` DESC, `transaction_ids.len()` DESC; **LIMIT 50**; `meta.truncated` when capped |

### 2. Route (frozen)

`GET /api/v1/subscriptions/discover?account_id=&payee=&interval_days=&amount_min=&amount_max=&limit=50`

Separate route from list API — Discover tab isolated; list API unchanged.

### 3. UI interval buckets

| Label | Target days |
|-------|-------------|
| Weekly | 7 |
| Biweekly | 14 |
| Monthly | 30 |
| Quarterly | 90 |
| Custom | user int |

**Account filter:** required in Discover form default (or first account pre-selected) — mitigates full-window scan without account.

### 4. Amount band

**P2 stretch** — implement when sprint capacity allows; not blocking AC-1 (payee + interval + account satisfy MVP).

## Alternatives rejected

| Alternative | Why rejected |
|-------------|--------------|
| Ad-hoc SQL `HAVING COUNT(*) >= 3` | Diverges from R-0009 cadence; DEC-0084/0086 drift risk |
| Re-run detection + filter pending only | Misses below-threshold merchants |
| Paginated discover results | 50-cap sufficient for household scale; simpler UX |

## Risks

- Large mirror without account filter — require account in UI; document in user guide
- &lt;3 txs false negatives — by design per R-0009
