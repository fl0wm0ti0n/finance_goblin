# DEC-0108 — Grafana provisioning-only fix: account variable default + mirror-count sync panel (BUG-0019)

**Status:** Accepted  
**Date:** 2026-06-10  
**Work item:** BUG-0019 (sub-defects **CA**/**CB**, acceptance **BG**, **BH**)  
**Research:** [R-0089](../docs/engineering/research.md#r-0089--bug-0019-grafana-cashflow-zeros-account_id-default--sync-entity-counts-per-run-cursor)  
**Extends:** **DEC-0107** (BUG-0018 — separate `alerts/evaluate.rs` read path, unchanged); R-0051 (`$forecast_variant` pattern, reused as precedent)  
**Supersedes:** DEC-0068 Y1 clause "Omit `current` block" for the `$account_id` variable — `current` is now required with empty text/value shape (`{"text": "", "value": ""}`); rest of DEC-0068 unchanged

## Context

Two confirmed root causes (R-0089, live DB probes + code audit):

- **CA (BG):** `$account_id` template variable in `cashflow.json` and
  `forecast-horizons.json` has `"sort": 1` (alphabetical), which re-sorts options
  **after** the SQL `ORDER BY ABS(COALESCE(a.balance,0)) DESC` — first option
  becomes **Cash wallet (116)** (all-zero forecast rows) instead of funded
  account **114** (731 non-zero rows in latest success). No `current` is set and
  the kiosk embed passes no `var-account_id`, so Grafana defaults to the first
  option. Data for 114 is non-zero — pure selection defect. Latent secondary:
  `cashflow.json` panels 1–3 pick latest success **without** `model_kind`
  filter; harmless today (only baseline succeeds) but diverges from the API the
  day `ml_enhanced` succeeds.
- **CB (BH):** `platform-health.json` panel 2 ("Records synced per entity")
  reads `sync_cursors.records_synced` verbatim, but `upsert_cursor` overwrites
  it with the **per-run** fetched-window count. Incremental transaction runs
  with 0 new transactions write `transactions: 0` while the mirror holds 922.
  Cursor semantics are correct for watermark bookkeeping; the panel semantics
  are wrong.

## Options

| Option | Verdict | Rationale |
|--------|---------|-----------|
| **CA-A — `sort: 0` on `$account_id` (both dashboards) + `current` default + `model_kind = 'baseline'` in cashflow latest-success subqueries** | **Accepted** | Pure provisioning change; SQL `ABS(balance) DESC` order holds → funded account first; no hardcoded account id; aligns panel computation choice with API default (R-0089 §3) |
| CA-B — embed forwards `var-account_id` from frontend | Rejected (deferrable) | New frontend↔Grafana coupling, needs account-source decision, doesn't fix dashboards opened directly in Grafana; can layer later as product feature |
| CA-C — household-level SUM panels | Rejected | Changes panel meaning; BG is per funded account; diverges from per-account API contract |
| **CB-A — panel SQL counts mirror tables directly (UNION ALL per entity) joined with `sync_cursors`** | **Accepted** | Matches panel title and BH literally; zero backend change; cursor/watermark semantics untouched; COUNT(*) trivial at this scale |
| CB-B — cumulative `records_synced` in `upsert_cursor` | Rejected | Unsound: overlap-days window re-fetches existing rows → systematic overcount; full-table entities inflate per run; needs migration/reset semantics |
| CB-C — backend-maintained `total_records` column | Rejected | Duplicates what live SQL computes; backend change + migration for a display concern |

## Decision

Accept **DEC-0108** as the BUG-0019 provisioning contract (no backend, frontend, or migration changes):

| Contract element | Value |
|------------------|-------|
| Files | `grafana/provisioning/dashboards/analytics/cashflow.json`, `grafana/provisioning/dashboards/analytics/forecast-horizons.json`, `grafana/provisioning/dashboards/platform-health.json` — **only** |
| CA variable fix | `$account_id` variable: `"sort": 0` in **both** cashflow and forecast-horizons; add `"current"` defaulting to the first (highest-\|balance\|) option so provisioned default is deterministic |
| CA subquery fix | `cashflow.json` panels 1–3: latest-success subquery gains `AND model_kind = 'baseline'` (forecast-horizons already filters via `$forecast_variant`, current=Baseline — unchanged) |
| CB panel fix | `platform-health.json` panel 2 `rawSql` replaced with mirror-count UNION ALL (sketch below); column `records_synced` may be retained only if relabeled "records last run" |
| Dashboard `version` | Bump in every edited JSON (cache-bust on re-provision) |
| Forbidden | Edits to `backend/src/db/mod.rs` `upsert_cursor`, `backend/src/firefly/mod.rs` sync semantics, `AnalyticsEmbedPage.tsx`, migrations; hardcoding account id `114` anywhere |
| Deploy step | Re-provision Grafana: `docker compose restart grafana` (or full redeploy); verify via kiosk embed **and** direct Grafana (covers no-`var-account_id` path) |

### CB panel SQL sketch

```sql
SELECT e.entity_type,
       e.records_total,
       c.last_successful_sync_at
FROM (
  SELECT 'transactions' AS entity_type, COUNT(*) AS records_total FROM transactions
  UNION ALL SELECT 'accounts',    COUNT(*) FROM accounts
  UNION ALL SELECT 'categories',  COUNT(*) FROM categories
  UNION ALL SELECT 'budgets',     COUNT(*) FROM budgets
  UNION ALL SELECT 'tags',        COUNT(*) FROM tags
  UNION ALL SELECT 'piggy_banks', COUNT(*) FROM piggy_banks
) e
LEFT JOIN sync_cursors c ON c.entity_type = e.entity_type
ORDER BY e.entity_type
```

(Entity list = the six `upsert_cursor` call sites in `backend/src/firefly/mod.rs`; mirror table names equal entity_type strings.)
