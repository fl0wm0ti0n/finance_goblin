# Decision DEC-0037

**Status:** Accepted  
**Date:** 2026-06-01  
**Story:** US-0007  
**Research:** R-0032, R-0035  
**Extends:** DEC-0004 (read-only connector pattern)

## Context

US-0007 requires Binance, Bybit, and Bitunix connectors importing balances, positions, trades, transfers, and funding. Discovery asked spot vs futures scope and read-only key enforcement.

## Options

1. **Unified `ExchangeConnector` trait; GET-only; spot + linear/USDT-M; Bitunix spot-first** — per R-0032
2. **CCXT library wrapper** — rejected (Rust ecosystem thin; hides PnL mapping)
3. **WebSocket-only ingestion** — rejected (needs REST backfill)
4. **Full derivatives parity day one including Bitunix futures** — rejected (signing complexity; spot delivers acceptance path)
5. **Per-exchange ad-hoc HTTP without trait** — rejected (duplicate backoff/signing logic)

## Recommendation

**Trait:** async `ExchangeConnector` with `test_connection`, `sync_balances`, `sync_positions`, `sync_trades`, `sync_transfers`, `sync_funding` normalizing to shared DTOs.

**Read-only:** GET-only HTTP layer + integration test rejecting write endpoints (DEC-0004 pattern). Operator keys must be read-only at creation.

**Scope:**
- Binance: spot + USD-M futures
- Bybit: UNIFIED account (spot + linear)
- Bitunix: spot MVP; futures stub behind `enabled_futures=false` default

**Sync:** per-symbol watermarks; 90-day first backfill + 1-day overlap; 429 exponential backoff; Binance `myTrades` limited to symbols with balance or open positions.

## Impact

- New `backend/src/exchanges/` module with three connector implementations
- `ExchangeService` orchestrates enabled connectors sequentially inside mutex

## Risks

- Binance per-symbol fan-out dominates sync latency — monitor and tune exchange interval
- Bitunix dual-domain signing increases test burden — dedicated integration test with mock server
