# DEC-0071 — Subscription alert dedup, unread-count contract, orphan lifecycle (BUG-0008 W)

**Status:** Accepted  
**Date:** 2026-06-08  
**Work item:** BUG-0008 (sub-defect **W**)  
**Research:** [R-0068](../docs/engineering/research.md#r-0068--bug-0008-subscription-alert-dedup-unread-count-contract-orphan-lifecycle), [R-0011](../docs/engineering/research.md#r-0011--subscription-price-change-detection--alert-thresholds), [R-0012](../docs/engineering/research.md#r-0012--subscription-persistence-schema-candidates-confirmed-rejections-events), [R-0023](../docs/engineering/research.md#r-0023--alert-persistence-deduplication--lifecycle-acknowledge--dismiss)  
**Extends:** DEC-0014 (alert types), DEC-0015 (subscription schema), US-0005 boundary (header bell)  
**Coordinate:** [R-0065 § BUG-0008](research.md#bug-0008-coordination-do-not-merge-scope) — additive unread-count route only; no `list_patterns` filter changes

## Context

Live probe (2026-06-08): **83** unread `new_detection` subscription alerts vs **6** pending patterns; operator intake **33 vs 11**. Root cause: `insert_alert` bare INSERT on every sync pass with no fingerprint dedup (`subscription_alerts` lacks partial unique index). Frontend `/subscriptions` banner derives count from raw alert list length — preserves mismatch after partial dedup.

Research [R-0068](research.md#r-0068--bug-0008-subscription-alert-dedup-unread-count-contract-orphan-lifecycle) resolved four W open questions. **W must ship before X recall work** (R-0069 §4) — threshold tuning without dedup re-amplifies alert spam.

## Decision

Freeze the **W bundle** as four contracts. Header bell remains **US-0005-only** (no subscription badge merge).

### 1. Lifecycle fingerprint dedup

| `alert_type` | Fingerprint | Re-fire rule |
|--------------|-------------|--------------|
| `new_detection` | `sub_alert:new_detection:{pattern_id}` | After mark-read, only if pattern returns to `pending` as a new row |
| `price_change` | `sub_alert:price_change:{pattern_id}:{direction}:{round(new_amount,2)}` | New episode after prior price alert read AND amount changes beyond R-0011 thresholds |
| `interval_change` | `sub_alert:interval_change:{pattern_id}:{interval_days}` | Same lifecycle as `price_change` |

**Schema migration:**

```sql
ALTER TABLE subscription_alerts ADD COLUMN fingerprint TEXT;
-- backfill + dedupe duplicates, then SET NOT NULL

CREATE UNIQUE INDEX subscription_alerts_unread_fingerprint
  ON subscription_alerts (fingerprint)
  WHERE read_at IS NULL;
```

**Insert contract** (`insert_alert` → `upsert_alert`):

```sql
INSERT INTO subscription_alerts (pattern_id, alert_type, title, body, sync_run_id, fingerprint)
VALUES ($1, $2, $3, $4, $5, $6)
ON CONFLICT (fingerprint) WHERE read_at IS NULL
DO UPDATE SET body = EXCLUDED.body, sync_run_id = EXCLUDED.sync_run_id, created_at = NOW();
```

**Pipeline gate:** call `upsert_alert` only when pattern upsert returns **new pending** OR confidence tier **increased** — skip when pending pattern unchanged on routine sync.

**One-time backfill:** group by fingerprint; keep newest unread per `(pattern_id, alert_type)`; mark-read duplicates (expect ~77 orphan duplicates from 83→6 steady state).

**Rejected alternatives:**

| Alternative | Why rejected |
|-------------|--------------|
| Per-`sync_run_id` dedup | Still ~6× pending mismatch across sync runs |
| Mark-read-only (no index) | Does not fix historical accumulation; races on concurrent sync |
| Merge into unified `alerts` table | R-0023 / US-0005 boundary |

### 2. Unread-count API contract

New route: **`GET /api/v1/subscriptions/alerts/unread-count`**

```json
{
  "unread_total": 2,
  "unread_new_detection": 2,
  "unread_price_change": 0,
  "pending_patterns": 6,
  "reconciled": true,
  "reconciliation_note": "unread_new_detection counts pending patterns with unread new_detection alerts; price_change alerts are informational"
}
```

**Semantics:**

- **Actionable unread** = unread `new_detection` alerts whose `pattern_id` references a **`pending`** pattern
- **`reconciled: true`** when `unread_new_detection <= pending_patterns` AND every unread `new_detection` joins to a pending pattern
- Steady state after dedup + orphan cleanup: **`unread_new_detection == pending_patterns`**

**UI surfaces consume this endpoint — not raw list length:**

| Surface | Source field |
|---------|--------------|
| `/subscriptions` banner | `unread_new_detection`; subtitle when `!= pending_patterns` |
| Post-sync toast | Fire only when `unread_new_detection` increases vs sessionStorage |
| Pending tab badge | `pending_patterns` (existing list or unread-count) |

**Trust metric:** `/subscriptions` banner + unread-count API — **not** header bell (unified count stays US-0005-only).

### 3. Orphan lifecycle hooks

| Event | Action |
|-------|--------|
| `POST .../confirm` | Mark-read unread `new_detection` for `pattern_id` |
| `POST .../reject` | Same; rejection fingerprint stops future `new_detection` |
| `mark_inactive` | Mark-read all unread alerts for pattern |

Apply same mark-read to `price_change` / `interval_change` when pattern terminal.
